# 深度审查：这个项目能跑通吗？

> **审查日期**：2026-07-14
> **审查方法**：对抗性深度审查 — 不逐行查错，只追问四个问题：可行吗？漏了什么？什么会坏？怎么排期？
> **前置阅读**：[第一次审查报告](2026-07-14-claude-md-and-plan-review.md)（逐节审查 + 一致性检查，本次不再重复）

---

## 一、核心判断

**结论先行：项目可交付，但不是 12 周。**

12 周计划有三类任务 — 骨架（M1/M6）、业务链路（M2/M3/M5）、模型微调（M4）— 其中模型微调是唯一的高风险不可控变量。如果 M4 的目标做务实取舍，11-14 周可完成一个能演示的 Demo；如果坚持所有指标达标，需要 16-20 周。

以下按风险从高到低逐一分析。

---

## 二、可行性分析（按风险排序）

### 🔴 风险 1：M4 微调效果 — 最大的不确定性

**问题**：计划的核心假设是"5k 标注样本 + Qwen2.5-1.5B QLoRA → F1 ≥ 0.85"。这个假设可能成立，也可能不成立，关键在于数据质量。

5k 样本来自 150 份年报 × ~33 个表格 = ~5000 个表格，每个表格提取 ground truth JSON。但 A 股年报的表格有这些特点：
- **科目名不一致**："货币资金"在不同公司可能叫"现金及银行存款"、"货币资金及存放中央银行款项"
- **层级嵌套**：合并报表和母公司报表混排，科目有缩进层级
- **单位混用**：同一张表可能"万元"和"元"混用（常见于银行/保险年报）
- **跨页表格**：一份资产负债表可能跨 3-4 页

PP-Structure 预填 + 人工校正的方式，校正一个表格至少需要 3-5 分钟。5000 个表格就是 **250-400 小时**的纯标注时间。两周不可能完成。

**建议**：
1. 标注量砍到 1500-2000 样本（从 150 份年报中精选）
2. 聚焦单一行业（如白酒/消费），减少科目名变体
3. 先用 7B 做 teacher model，自动标注大部分数据，人工只审核不一致的
4. F1 目标从 0.85 降为 0.75（对齐 M2 的 0.70 基线，微调只需提升 0.05）

---

### 🔴 风险 2：Windows 兼容性 — 被低估的系统性风险

**问题**：CLAUDE.md 和设计文档都没有明确说明核心组件在 Windows 上的兼容性。当前开发环境是 Windows 11。以下是已知问题：

| 组件 | Windows 状态 | 严重度 |
|---|---|---|
| bitsandbytes（4-bit 量化） | 仅支持 Linux/WSL2，原生 Windows 不支持 | 阻塞 |
| llama.cpp | Windows 有预编译 binary，可用 | ✅ |
| WeasyPrint | Windows 需要额外 GTK+ 依赖 | 中 |
| PP-StructureV2 | 依赖 PaddlePaddle，Windows 支持但不稳定 | 中 |
| Milvus 2.4 | Docker 运行，不受影响 | ✅ |
| Flash-Attention | Windows 不支持 | 不影响（6GB VRAM 本就不该用） |

**最关键的是 bitsandbytes**：4-bit QLoRA 训练在原生 Windows 上不可行。要么 WSL2，要么用替代方案（llama.cpp GGUF 量化 + 纯 CPU offload）。**建议在开工前立即验证 bitsandbytes 在 WSL2/原生 Windows 上的可用性。**

---

### 🟡 风险 3：8 个 Docker 容器的内存压力

**问题**：笔记本跑 8 个容器 + 模型推理，内存够吗？

粗略估算：
- MySQL 8.0: ~500MB
- Redis 7: ~100MB
- Milvus 2.4 (standalone): ~1GB
- RabbitMQ 3.13: ~300MB
- MinIO: ~200MB
- backend (JVM): ~500MB
- ai-service (Python + 模型): ~5.5GB（7B 4bit ~5GB + runtime）
- frontend (Node): ~200MB
- Windows 系统: ~3GB
- **合计**: ~11GB

如果笔记本 16GB RAM，勉强够用。但加上 IDE（VS Code ~1GB）和浏览器，就会触发 SWAP，导致推理速度暴跌。

**建议**：
- 开发时只启动依赖容器（mysql/redis/rabbitmq/minio，去掉 milvus）
- 模型推理走本地进程（不容器化 ai-service）
- Milvus 按需启动（M5 之前不需要）

---

### 🟡 风险 4：设计复杂度与时间压力

**问题**：76 个任务，12 周，每周约 6.3 个任务。看起来合理，但：

1. **M1 的 17 个任务涉及 3 种语言、8 个服务**。这不是单纯的"骨架"，而是需要同时处理 Java/Spring、Python/FastAPI、Vue3/TypeScript 三套技术栈。跨栈上下文切换成本很高。

2. **每个任务要求 80% 测试覆盖率 + CI**。这在个人项目中极其耗时 — 写测试的时间和写代码的时间至少 1:1。

3. **MQ + SSE 混合架构的 debug 难度远高于同步调用**。消息丢了、顺序乱了、SSE 断了 — 这些问题在早期很难定位。

**现实预估**：如果每周投入 50h（非全职 60h），12 周约 600h。扣除学习/调试/休息，可用约 450h。76 个任务平均每个 6h，但模型训练任务（M4）一个就要 9h 纯计算 + 数小时调试。所以时间确实紧。

**建议**：区分"核心路径"和"锦上添花"。M1 的 mock 链路是核心路径，但 CI（M1.17）可以推迟到 M2 再做；Prometheus/Grafana/Jaeger（M6.05-07）在 Demo 阶段可以不做。

---

### 🟢 风险 5：PDF 解析质量 — 有兜底方案，可控

**问题**：A 股年报 PDF 质量参差不齐。但设计文档有 OCR 兜底（M2.03），表格识别有 LayoutLM 微调（M4），风险可控。

**额外建议**：在 M2.01 实现时，先测试 10 份不同公司的年报，统计 PyMuPDF 直接提取文本的成功率和表格还原率。这决定了 M2 后续任务是否需要更多精力在预处理上。

## 三、遗漏点分析

### 遗漏 1：任务状态机的精确定义

**问题**：设计文档 §3.2 和 CLAUDE.md 都提到"任务状态机"和"状态转换"，但没有给出完整的状态机图。以下状态转换需要明确定义：

```
PENDING → PARSE_RUNNING → PARSE_SUCCESS → EXTRACT_RUNNING(×3) → EXTRACT_SUCCESS(×3)
→ CHECK_RUNNING → CHECK_SUCCESS → REPORT_RUNNING → REPORT_SUCCESS → COMPLETED

任何阶段：*_RUNNING → *_FAILED → RETRYING → *_{SUCCESS|FINAL_FAILURE}
```

关键问题：
- EXTRACT 三条中有一条失败，是否回滚已成功的两条？文档说"整任务 FAILED"——但已写入 financial_statement 的数据如何处理？
- 用户取消任务时，正在执行的 L3 worker 怎么通知？
- 超时重试的指数退避参数是什么（1s → 2s → 4s？还是 10s → 30s → 60s？）

**建议**：在 M1.09 编码前，画一张完整的状态机图（可以用 Mermaid），贴在设计文档里。

---

### 遗漏 2：API 返回值的具体定义

**问题**：设计文档 §6 列出了 25 个 API 端点和几个关键接口的请求/响应格式（如 upload、SSE 事件），但大部分接口只有路径没有响应格式。以下接口的响应格式需要提前定义：

| 接口 | 缺失内容 |
|---|---|
| `GET /reports` | 分页参数（page/size？cursor-based？）、列表字段 |
| `GET /reports/{id}` | 响应体包含哪些字段？是否嵌套 statements/checks？ |
| `GET /reports/{id}/statements` | 返回的 JSON 结构与 M7 抽取格式一致还是简化的？ |
| `POST /chat/sessions/{id}/messages` | 请求体除了 content 还有哪些字段（temperature？max_tokens？） |

**建议**：不需要一次性写完所有 API 的完整定义，但至少在开始对应任务前，把该任务涉及的接口响应格式定下来。

---

### 遗漏 3：前端的"可编辑表格"需求没定义清楚

**问题**：M2.11 提到"可编辑表格组件"和"可手动编辑数值（暂不写回）"——但"编辑"的目标是什么？

- 如果是为了让用户修正 AI 抽取的错误 → 这是数据标注/反馈回路的基础设施
- 如果只是为了让用户能查看大数值 → 那不需要"编辑"，只需要好的展示
- "暂不写回"说明当前没有持久化编辑的意图 → 那为什么要做成可编辑？

**建议**：明确 M2.11 的目标是"只读展示 + 排序/筛选"，编辑功能留到 M6 或砍掉。一个好看的三表展示比一个功能不全的编辑表格更有演示价值。

---

### 遗漏 4：openapi.yaml 与实际 API 的同步机制

**问题**：设计文档 §6.8 说 L2 用 springdoc-openapi 自动生成 OpenAPI 3.1，同时又说接口变更必须同步更新 `docs/api/openapi.yaml`。这两个文件的关系没说清楚：
- `openapi.yaml` 是自动生成然后手动编辑的？
- 还是手写 yaml，springdoc 只做参考？

**建议**：以 springdoc 自动生成为准，`docs/api/openapi.yaml` 作为导出归档（在 M6.10 时从 springdoc 导出），平时不维护。

---

### 遗漏 5：对话系统中的"多财报"场景

**问题**：设计文档 §6.2.4 的 chat_session 关联一个 report_id，每个会话绑定一份财报。但用户可能问："比较茅台和五粮液的毛利率"——这需要跨财报查询。

当前设计不支持跨财报会话。如果在 M5 后期才发现这个需求，改动成本很高（需要改 chat_session 表结构、M9 工具注册、上下文管理）。

**建议**：M5 的 chat_session 改为关联多个 report_id（用 JSON 数组或中间表），支持多财报问答。如果考虑到开发量，可以在 M5 阶段先限制单财报问答，设计上预留扩展点。

## 四、架构压力测试

### 场景 1：用户在任务执行中关闭浏览器

1. 前端 SSE 连接断开
2. L2 SseEmitterPool 清理该 emitter → ✅ 正常
3. L3 worker 继续执行 → 完成后发 progress 消息 → L2 收到后发现无 emitter → 丢弃
4. 用户重新打开页面 → 没有 SSE 连接，看不到旧的 progress

**问题**：用户重开后看不到已完成步骤的状态。设计文档提到 `Last-Event-ID` 和 Redis 进度恢复，但这只在 M1.11 中做了基础实现。需要确认：重连时 L2 从 Redis/MySQL 读取当前进度，立即推送一次完整的进度快照。

---

### 场景 2：7B 模型推理时收到 EXTRACT 请求

1. REASON 链路正在用 7B（显存占用 ~5GB）
2. EXTRACT 请求到达 → model_lock 被 7B 持有 → 尝试 nack(requeue=true, delay=2s)
3. 2s 后消息回到队列 → 再次尝试 → 如果 7B 还在推理（可能跑 30s+）

**问题**：nack 重试会无限循环直到 7B 释放。但 RabbitMQ 的 nack(requeue=true) 没有内置 delay——消息立即回到队列头部，消费者很快再次拉取（prefetch=1 的消费者在 ack/nack 后会取下一帧）。这会导致**忙等**，白白消耗 CPU 和 RabbitMQ 往返。

**建议**：用死信 + TTL 实现延迟重试（消息 nack(requeue=false) → 进入一个带 5s TTL 的延迟队列 → 超时后投递回原队列），或直接用 Redis 的阻塞队列替代 MQ 等待。

---

### 场景 3：EXTRACT 三表并行但有一张表抽取返回空

1. BS 表正常，IS 表正常，CF 表只返回了 2 个科目（实际应该有 30+）
2. Progress 事件仍然返回 SUCCESS（因为 JSON 解析成功）
3. L2 收到 3 条 SUCCESS → 触发 CHECK

**问题**：CHECK 发现 CF 数据严重缺失，但此时已经进入 CHECK 阶段。按照"三表都 SUCCESS 才触发 CHECK"的逻辑，CHECK 本身无法判断数据是否完整。

**建议**：EXTRACT 完成后 L3 返回一个 `item_count` 字段，L2 校验每张表至少 > 20 条记录才认为成功。加一个 confidence 阈值判断。

---

### 场景 4：模型降级链的延迟累积

1. 本地 7B 超时 60s → 切 API 72B
2. API 72B 调用 + 网络延迟 ~10s
3. M9 ReAct 循环可能调用 7B 3-5 次

如果每次 ReAct 步骤都超时再切 API，单次问答可能从预期的 30s 膨胀到 60+3×(60+10)=270s。

**建议**：降级一旦触发（第一次超时），后续该 task 的所有推理直接走 API，不再尝试本地 7B。用 Redis 记录 `fin:fallback:{taskId}` 标志位。

## 五、执行策略建议

### 当前计划的并行策略有待优化

计划 §9.2 的并行策略：
```
M2 (解析抽取) || M4.01-02 (数据采集 + 标注)
M3 (勾稽报告) || M4.06 (T2 数据构造)
M3 末期       || M5.01-03 (ReAct 框架)
```

问题：早期并行力度不够，M1/M2 串行完成后才发现训练数据不够，损失 4 周。

### 建议的执行策略：早验证、晚优化

| 周 | 主线 | 并行 | 验证点 |
|---|---|---|---|
| W1 | M1 基础设施 | 下载 3 份测试 PDF + 下载 GGUF 模型 | bitsandbytes 在 WSL2 可跑 ✅/❌ |
| W2 | M1 骨架 | 用 10 份年报测试 PP-Structure 解析率 | 表格还原率 > 60% ✅/❌ |
| W3-4 | M2 解析抽取 | T1 数据标注启动（每天 2h 标注） | 7B 通用 prompt 抽取 F1 |
| W5-6 | M3 勾稽报告 | T1 标注继续 + T2 数据构造 | 勾稽准确率 |
| W7 | M4 T1 训练 | M3 收尾 | 1.5B 首个 epoch 的 val loss |
| W8 | M4 T2/T3 训练 | M5.01-03 ReAct 框架 | — |
| W9-10 | M5 Agent 问答 | — | 问答命中率 |
| W11-12 | M6 打磨发布 | — | 端到端评估 |

关键变化：**标注提前到 W3 开始**，与 M2 并行。如果 W7 训练效果差，还有 W8 一整周补救。

---

### 建议的 MVP 策略：如果时间不够

如果到 W8 发现 M4 效果不达预期，可以做出以下取舍：

1. **砍掉 T3 LayoutLM**：用 PP-Structure 原版表格识别 + 正则兜底。T3 改善的是表格还原的美观度，不直接影响抽取 F1。
2. **砍掉 T2 微调 embedder**：bge-small-zh 原版在中文上的 MRR@10 约 0.65-0.70，微调提升到 0.78 对 Demo 演示价值有限。
3. **保留 T1**：1.5B QLoRA 是唯一不能砍的 — 它决定了整个系统的抽取质量。
4. **砍掉 M9 ReAct 可视化的 thought/tool_call 折叠面板**：普通聊天气泡 + Markdown 渲染就足够演示。

如果只保留 T1，M4 工作量从 15 个任务降为 ~8 个任务，省约 50h。

---

### 建议的加载策略：模型预加载

设计文档没有明确模型加载的时机。对于 Demo，建议：

1. **Day 1 启动时**：脚本预加载 7B 到显存（`/internal/models/load`），此后常驻
2. **冷启动 SLA**：首次调用 > 30s 是正常的（包括 FastAPI 启动 + 模型加载），不必追求 15s
3. **热启动 SLA**：模型已加载后，7B 推理首 token 约 3-5s（100 token prompt），报告生成约 20-30s（1500 token 输出）

## 六、文档维护建议

### CLAUDE.md 应该分开为两个文件

当前 CLAUDE.md 承担了两个职责：
1. **AI 协作规范**（给 AI 看的：代码规范、模板、禁止事项）
2. **项目配置参考**（给人看的：版本号、目录结构、命令列表）

建议拆分为：
- `CLAUDE.md`：纯 AI 协作规范（保持当前 §3/§9/§11），每次会话都读
- `docs/DEVELOPMENT.md`：技术手册（当前 §1/§2/§7/§8/§12/§15），需要时翻阅

这样 AI 每次加载 CLAUDE.md 的速度更快，且不会因为版本号变更污染 AI 行为。
