## 背景

实现 spec §3.9 显存调度：6GB VRAM 单进程只能装 1 个 7B(4bit ~5GB) 或 1 个 1.5B + 1 个 small，多 worker 进程会同时加载冲突模型；用 Redis 分布式锁 `fin:lock:model:{modelName}` 协调，TTL 5min（spec §5.4.1）。本任务依赖 M2.04 已完成的 `ModelHub` / `LlmLoader` 单模型驻存管理。

任务卡（plan §4 M2.05）：
- 涉及文件：`ai-service/app/modules/modelhub/vram_scheduler.py` + `ai-service/app/core/redis_client.py`
- 验收标准：多 worker 不会同时加载 7B；切换模型时正确 unload
- 验证方式：起 2 个 worker 并发请求，检查只有 1 个加载 7B

## 决策列表

### D1: 不引入 fakeredis，测试内置 _FakeRedis

**初始问题**：单测需要 mock Redis；社区惯例是用 `fakeredis`，但 AGENTS.md §9.2 明确"禁止引入设计文档未提及的新依赖"，spec §1 技术栈表也未列 fakeredis。

**决策**：在 `tests/test_m2_vram_scheduler.py` 内置 `_FakeRedis` 类，实现 `RedisLike` Protocol 表面的最小集：
- `set(name, value, *, ex, nx)`：SET NX EX 语义
- `get(name)` / `delete(name)` / `expire(name, ex)`
- `eval(script, numkeys, *args)`：内嵌 Lua 解释器，识别 release/refresh 两种脚本（按 `"del"` / `"expire"` 关键字分支）
- `ping()` + `fail_ping` 开关
- `_ttl: dict[str, float]` 跟踪 TTL，`_expired(key)` 检查过期

`_FakeRedis` ~130 行，覆盖 ModelLock + VramScheduler 所有路径。后续 M3+ 的 Redis 测试可复用此 stub（提取到 `tests/_fakes/` 即可）。

### D2: 非重入锁 + Lua owner 校验（canonical Redis lock pattern）

**初始问题**：spec §3.9 只说"获取 lock"，但生产级 Redis 锁需要解决：
- 两个 worker 同时 SET，NX 保证只有一个成功
- Worker A 持锁，TTL 到期自动释放；Worker B 抢到锁；Worker A 完成 `generate()` 后 DEL 会误删 B 的锁
- 同理 EXPIRE 也会被 A 误续 B 的锁

**决策**：`ModelLock` 采用 Redis 官方推荐模式：
- `acquire()`：`SET fin:lock:model:{name} {worker_id} NX EX {ttl}`，原子获取，无需 GET+SET 双调用
- `release()`：Lua 脚本 `if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('del', KEYS[1]) else return 0 end`，仅当 stored value == worker_id 时 DEL
- `refresh()`：同理用 Lua 校验 owner 后 EXPIRE
- `worker_id` 默认 `uuid.uuid4()`，单 worker 进程内所有 `ModelLock` 共享（在 `VramScheduler` 里通过 `self._worker_id` 注入）；多 worker 进程用不同 uuid 天然隔离

### D3: load_for_scene_with_lock 不自动 release，返回 ModelLock 给调用方

**初始问题**：spec §3.9 说"消费消息前先获取对应模型的 lock"——但 lock 应该覆盖整个 `generate()` 推理窗口，不只是 `load()`。如果 `load_for_scene_with_lock` 在内部 release，generate() 期间另一 worker 可以抢锁切到别的模型，把当前模型的显存卸掉。

**决策**：`load_for_scene_with_lock(scene)` 返回 `ModelLock` 实例，调用方负责 release：
```python
lock = sched.load_for_scene_with_lock(Scene.REASON)
try:
    result = hub.generate(prompt, ...)
finally:
    lock.release()
```
或者用 context manager：
```python
with sched.load_for_scene_with_lock(Scene.REASON) as lock:
    result = hub.generate(prompt, ...)
```

`load_for_scene_with_lock` 内部用 try/except 兜底：`hub.load_for_scene(scene)` 失败时自动 `lock.release()` 防泄漏。

### D4: LRU evict_idle 只卸载驻存模型，stale entry 静默 pop

**初始问题**：`_last_used: dict[str, float]` 跟踪每个模型最后使用时间。但 `LlmLoader` 单模型驻存意味着任何时刻只有一个模型在显存里；`_last_used` 里可能有"曾经驻存但已被切走"的旧 entry。evict_idle 如果盲目 `hub.unload()` 会卸错模型。

**决策**：`evict_idle(threshold)` 遍历 `_last_used`：
- `loaded_model() == model_key`（当前驻存）且 `now - last_used >= threshold` → `hub.unload()` + pop entry
- `loaded_model() != model_key`（已被切走）→ 静默 pop entry（不调用 unload，避免误卸当前模型）
- `loaded_model() == model_key` 且 `now - last_used < threshold` → 跳过

M2.05 不接后台 tick；spec §3.9 描述的"避免显存抖动"靠场景路由表（EXTRACT 链路只用 1.5B、REASON 链路只用 7B）+ 单模型驻存 no-op 已经达到。`evict_idle` 留给 M3/M5 的定时任务或 MQ 空闲槽位触发，本任务只交付 API。

### D5: RedisClient 包装层接受 RedisLike Protocol

**初始问题**：`VramScheduler` 需要一个 Redis 客户端；测试想直接注入 `_FakeRedis`，生产想用 `redis.Redis.from_url(...)`。如果类型注解为 `redis.Redis`，测试 stub 无法通过类型检查。

**决策**：
- 定义 `RedisLike` Protocol（`set` / `get` / `delete` / `expire` / `eval` / `ping` 6 个方法）
- `RedisClient` 包装 `redis.Redis`，懒加载，`client` property 返回 `RedisLike`
- `VramScheduler(redis_client: RedisClient | RedisLike | None)`：
  - `None` → 用 `get_redis_client().client`（生产路径）
  - `RedisClient` 实例 → 自动 unwrap `.client`
  - 任意 `RedisLike`（如 `_FakeRedis`） → 直接用

这样生产代码用 `RedisClient` 抽象，测试代码用裸 `_FakeRedis`，类型注解都通过。

### D6: RedisClient.ping 失败转 False 不抛

**初始问题**：M1 的 `/internal/health` readiness 探针要检测 Redis；如果 Redis 不可用时抛异常，health endpoint 会 500，K8s readiness 会重启 pod 而不是标记 not-ready。

**决策**：`RedisClient.ping()` 用 try/except 包裹，失败时 `LOGGER.warning` + 返回 False；调用方按 False 走 not-ready 分支。`RedisClient.connect()` 仍会抛 ImportError（缺 redis 包是部署错误，不是运行时故障）。

### D7: URL 凭据脱敏日志

**初始问题**：spec §11.1 日志规范要求脱敏 password / token 字段；`redis_url` 可能含 `redis://:secret@host:6379/0`，直接 LOGGER.info 会泄漏密码。

**决策**：`RedisClient._mask_url(url)` 用 `urllib.parse` 解析，把 password 替换为 `***`，保留 user/host/port/db 用于排障：
```
redis://:***@host:6379/0
```
`connect()` 日志用 `self._mask_url(self.settings.redis_url)` 而非裸 URL。

### D8: 未集成 MQ consumer，留给 M2.06+

**初始问题**：spec §3.9 说"消费消息前先获取对应模型的 lock"，听起来 M2.05 应该改 `app/mq/consumer.py`。但 `TaskConsumer` 已经在 M1.14 稳定（覆盖 retry / DLQ / progress / ack 全套测试），改它要回滚大量测试。

**决策**：M2.05 只交付 `ModelLock` + `VramScheduler` 核心组件 + 单测，不动 `app/mq/consumer.py`。M2.06+ 的 extract handler 在消费 `q.extract.requests` 时显式调用：
```python
def handle(task: TaskMessage) -> dict:
    sched = get_vram_scheduler()  # M2.06 引入
    try:
        lock = sched.load_for_scene_with_lock(Scene.EXTRACT)
    except ModelLockBusyException:
        time.sleep(settings.model_lock_retry_seconds)
        raise  # let consumer nack(requeue=True)
    with lock:
        return _do_extract(task)
```
这样 M1.14 的 MQ 链路完全不变，M2.05 的核心组件通过 handler 接入。AGENTS.md §9.2 "一次只解决一个问题" + §14.1 任务级 DoD 也支持这种增量。

### D9: Windows time.monotonic() 精度问题

**初始问题**：`test_lock_refresh_extends_ttl_when_owner_matches` 用 `time.sleep(0.01)` 后比较新旧 TTL；Windows 上 `time.monotonic()` 精度约 15ms，0.01s 可能被四舍五入到同一 tick，导致 `new_ttl == original_ttl`。

**决策**：测试用 `time.sleep(0.05)` 跨过 Windows 粒度，注释说明原因。生产代码不依赖此精度（TTL 5min 远大于 15ms）。

## 发现的风险

1. **未做多 worker 集成验证**：spec §3.9 要求"起 2 个 worker 并发请求，检查只有 1 个加载 7B"。本次只跑单进程 _FakeRedis 单测，真实双 worker 端到端验证留到 M2.06+ 的 Extractor handler 接入后用 docker-compose 起两个 ai-service 容器跑。需要确认：(a) 两个 worker 进程的 uuid 真的互斥；(b) `SET NX EX` 在 Redis 7.2 上原子性如预期；(c) TTL 5min 够长，单次 7B 推理（spec SLA 60s）不会触 TTL 自动释放
2. **lock TTL 5min vs 推理 SLA 60s 的缓冲不够大**：spec §5.4.1 TTL=5min，spec §12.1 单次 generate SLA=60s。如果 worker 卡死（OOM/GIL），TTL 到期后另一 worker 抢锁切模型，但原 worker 的 daemon 线程可能仍在跑 `model.generate()` 占显存，导致切模型 OOM。spec §10.3 "本地 7B OOM → 释放模型 → 重启 ModelHub" 需要进程级重启，M2.05 的 `ModelLock` 没有重启 ModelHub 的能力
3. **Redis 不可用时 ModelLock 抛 LockAcquisitionException**：`SET` 失败时 acquire 抛 `LockAcquisitionException`，调用方需要决定降级策略。spec §10.3 没明确 Redis 不可用时的降级；M2.06+ 的 handler 应捕获此异常转 `nack(requeue=True)` + 退避，避免 Redis 抖动时整个 EXTRACT 链路 500
4. **_FakeRedis 的 Lua 解释器是简化版**：只识别 release/refresh 两种脚本（按 `"del"` / `"expire"` 关键字），不支持通用 Lua。如果未来 M3+ 引入其他 Lua 脚本（如 SETNX + EXPIRE 合并、原子 INCR），需要扩展 _FakeRedis 或迁移到 fakeredis（届时需提变更申请）
5. **VramScheduler 单例未实现**：M2.05 只交付类，未提供 `get_vram_scheduler()` 模块级单例。M2.06 的 Extractor handler 需要单例时，要么自己缓存，要么补 `get_vram_scheduler()` 函数（建议放 `app/modules/modelhub/__init__.py`，与 `get_modelhub()` 对称）
