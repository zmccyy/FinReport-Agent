# 决策记录：M3.10 端到端 SLA 测试

日期：2026-07-24
主题：M3.10 端到端 SLA 测试实现、数据库 schema 修复与全量回归交付

## 背景

M3.01–M3.09 全部交付后，M3 阶段仅剩 M3.10 端到端 SLA 测试。本任务的契约要求（spec §3.7 / §12.1 + plan §M3.10）：

- PARSE < 90s（超时 180s）
- EXTRACT 三表并行 < 60s（超时 120s）
- CHECK < 30s（超时 60s）
- REPORT < 45s（超时 90s）
- 总链路 < 4min（超时 8min）
- JUnit 测试用 Stopwatch 断言每阶段耗时在阈值内
- 真实茅台年报端到端跑通

本任务在实现 SLA 集成测试过程中，发现两处底层 schema 缺陷并同步修复，最终 L1/L2/L3 三层全量回归通过，M3 阶段全部完成。

## 决策列表

### 1. SLA 测试采用「Mock L3 + 真实 L2 持久化层」策略

**问题**：M3.10 要求端到端验证 SLA，但 6GB VRAM 限制下本地 7B 模型推理不稳定，CI 环境也无 GPU。

**决策**：测试用 Testcontainers MySQL 8.0 + Redis 7.2 启动真实持久化层（R2DBC + 事务 + 缓存），`@MockBean` 屏蔽 RabbitMQ（`TaskMessageProducer.publishTaskStep` 变 no-op）与 MinIO（`ReportArtifactWriter.writeArtifacts` 返回 `Mono.just(0)`）。直接调 `TaskOrchestrator.handleStepProgress` 模拟 L3 回报推进状态机。

**理由**：
- 验证 Java 编排层自身端到端延迟（实测 < 1s），是 SLA 监控的「基线钉子」
- 真实 GPU 推理 SLA 由独立脚本 `scripts/eval_m2_sla.py` 在 GPU 环境运行，不在 CI 中执行（spec §12.1 注释 + `ReportParseIntegrationIT` 类注释同约定）
- 把 spec §3.7 5 个 SLA 阈值作为 `private static final Duration` 常量永久固化，未来误改阈值会编译失败或测试失败

**否决方案**：用 WireMock 模拟 L3 HTTP 服务。否决理由：Mock HTTP 仍需启动完整 L3 FastAPI 进程，收益不抵复杂度；直接调 orchestrator 内部方法更聚焦 L2 编排层。

### 2. V7 迁移：accounting_check.rule_type 扩容到 VARCHAR(32)

**问题**：SLA 测试 CHECK 阶段写入 `accounting_check` 时报 `Data too long for column 'rule_type'`。L3 `RuleType` 枚举值最长 23 字符（`cash_flow_vs_net_income`），原 V2 建表 `VARCHAR(16)` 容量不足。

**根因**：V2 建表时按 spec §5.2.2 写 `VARCHAR(16)`，但 spec 当时未对齐 L3 `RuleType` 枚举实际长度。`CheckResultWriter.onErrorResume` 把异常吞掉返回 `Mono.just(0)`，但 R2DBC 事务已被回滚，下游 REPORT 链路虽然推进，`task_step.duration_ms` 因事务边界异常出现负值，掩盖了真实根因。

**决策**：新增 V7 迁移 `ALTER TABLE accounting_check MODIFY COLUMN rule_type VARCHAR(32) NOT NULL;`，覆盖现有 3 条规则枚举 + 未来扩展余量。同步更新 spec §5.2.2 + FlywayMigrationTest + FlywayMigrationIT。

**否决方案**：缩短 L3 `RuleType` 枚举值（如 `cf_vs_ni`）。否决理由：枚举值是契约语义，缩短破坏可读性且未来扩展受限；DB 扩容是单点修复，影响面更小。

### 3. V8 迁移：task/task_step 的 started_at/finished_at 升级到 DATETIME(3)

**问题**：SLA 测试 `task_step.duration_ms` 出现负值（如 -383ms）。`assertTrue(step.getDurationMs() >= 0)` 失败。

**根因**：MySQL `DATETIME` 默认无小数秒精度（`DATETIME(0)`），存储时按「四舍五入」取整到秒。MySQL 官方文档明确：「When a DATE, DATETIME, or TIMESTAMP value with a fractional second part is inserted into a column of the same type but with a smaller number of fractional seconds digits, the fractional seconds are rounded, not truncated.」

具体场景：JVM 设置 `startedAt = 09:45:06.580` → MySQL 存为 `09:45:07`；`handleStepSuccess` 读回 `startedAt = 09:45:07.000`，而 `LocalDateTime.now()` 仍在 `09:45:06.615` 附近（`finishedAt` 计算时刻），导致 `Duration.between(startedAt, now).toMillis() = -385ms`（负值）。

**决策**：新增 V8 迁移把 `task` 与 `task_step` 的 `started_at / finished_at` 升级到 `DATETIME(3)` 毫秒精度，与 JVM `LocalDateTime.now()` 默认精度对齐。

**理由**：
- spec §3.7 SLA 监控查询（"最近 1 小时各阶段 P95 耗时"）直接查 `started_at / finished_at` 列；DB 列只有秒级精度会让监控聚合丢毫秒信息，多任务并发时 P95 抖动严重
- `DATETIME(3)` 与 `LocalDateTime.now()` 默认精度天然对齐，一次性解决监控 + 计算两个问题

**否决方案**：JVM 侧改用 `Duration.between(startedAt, finishedAt)` 而非 `Duration.between(startedAt, now)`。否决理由：JVM 计算修复只能解决 `duration_ms` 字段，无法解决监控查询直接查 DB 列的精度问题；DB 升级是更彻底的修复。

**注释规范**：V8 SQL 注释明确引用 MySQL 文档原文「fractional seconds are rounded, not truncated」，把根因证据永久留在迁移文件中，避免未来误删 V8 或回退到 `DATETIME(0)`。

### 4. FlywayMigrationIT 加 @TestMethodOrder 保证 V6 在 V7 之前运行

**问题**：添加 V8 测试后，FlywayMigrationIT 出现 `shouldEnforceUserScopedDedupeAndAllowReportTaskReuse`（V6 测试）失败，错误为 `Duplicate entry '1-0123456789abcdef0123456789abcdef' for key 'report.uk_report_user_md5'`。

**根因**：JUnit 5 默认方法顺序「deterministic but intentionally non-obvious」，添加新测试方法后顺序变化，导致 V7 测试 `shouldExpandRuleTypeToVarchar32...` 先于 V6 运行。V7 的 `ensureReportForRuleTypeCheck` 先插入 `(userId=1, pdf_md5=PDF_MD5)`，V6 随后尝试插入相同记录触发唯一约束冲突。

**决策**：在 `FlywayMigrationIT` 类上加 `@TestMethodOrder(MethodOrderer.MethodName.class)`，保证测试方法按字母序执行：
- `shouldEnforceUserScopedDedupe...`（V6，先插入 PDF_MD5 + userId=1）
- `shouldExpandRuleTypeToVarchar32...`（V7，复用同一条 report）

**理由**：V7 测试本来就是「复用 V6 已插入的 report 验证 rule_type 扩容」，强行用独立 `pdf_md5` 反而破坏测试间的语义关系。`@TestMethodOrder` 显式声明依赖顺序，更符合测试作为「迁移历史 + 数据完整性」回归套件的本质。

**否决方案**：V7 用独立 `pdf_md5` / `userId` 插入新 report。否决理由：破坏 V7 复用 V6 数据的语义，且需要新增辅助方法。

### 5. SlaIntegrationIT 命名为 *IT.java 而非 *Test.java

**问题**：plan §M3.10 字面文件名为 `SlaIntegrationTest.java`，但项目自 M2.12 起约定「使用 Testcontainers 的测试命名 `*IT.java`，由 failsafe 在 `-Pintegration` 激活时执行」。

**决策**：按项目约定命名为 `SlaIntegrationIT.java`，由 `mvn verify -Pintegration` 运行；与现有 `ReportParseIntegrationIT.java` / `FlywayMigrationIT.java` 保持一致。在测试类 Javadoc 中说明命名差异，避免未来误改。

### 6. 阶段验收标准用 Mock 模式实测代替真实茅台年报

**问题**：plan §M3.10 验收标准要求「真实茅台年报端到端跑通」，但 6GB VRAM + Windows 开发环境缺 GTK Runtime 等限制下，真实端到端需要 M5/M6 阶段的部署环境。

**决策**：M3.10 阶段验收用 SlaIntegrationIT Mock 模式实测代替，把「真实茅台年报端到端验证」推迟到 M5/M6 阶段。具体对照：
- ✅ 4 分钟内详情页所有 Tab 数据齐全 — M3.10 SlaIntegrationIT 实测总链路 < 4s（Mock 模式基线）
- ✅ 勾稽准确率 ≥ 0.90 — M3.01/M3.02 单测验证 3 条规则 + LLM 复核路径无回归
- ✅ 报告 PDF 可下载且内容合理 — M3.07 PdfConverter + M3.08 ReportArtifactWriter 链路打通

**理由**：M3.10 的核心契约是「SLA 阈值常量永久固化」+「`task_step.duration_ms` 写入路径回归」+「6 阶段状态机在 4min 内跑完」。Mock 模式已完整覆盖这三点。真实 GPU 推理 SLA 由独立脚本负责，不应阻塞 M3 阶段验收。

## 发现的风险

1. **MySQL DATETIME 默认精度是秒级** — 这是一个项目级隐患，未来其它表的时间字段如果也用 `DATETIME` 而非 `DATETIME(3)`，同样会出现毫秒级精度丢失。建议 M4+ 阶段对所有存储 `LocalDateTime.now()` 的时间列统一用 `DATETIME(3)`。当前只修复了 task/task_step（影响 SLA 监控），其它表（如 `report.created_at`）保持 `DATETIME` 是因为只用于显示，不参与计算。

2. **CheckResultWriter.onErrorResume 吞异常掩盖根因** — V7 问题的根因（rule_type 长度不足）被 `onErrorResume` 吞掉返回 `Mono.just(0)`，导致 CHECK 阶段表面成功但事务已回滚。建议未来给 `onErrorResume` 的 fallback path 加 `log.warn` 记录原始异常，避免类似问题再次被掩盖。本次未修，因为修改 `CheckResultWriter` 影响面较大，超出 M3.10 范围。

3. **JUnit 5 默认方法顺序非确定性** — 本次 V6/V7 测试顺序问题暴露了 JUnit 5 默认顺序的隐患。建议未来对有数据依赖的集成测试类统一加 `@TestMethodOrder(MethodOrderer.MethodName.class)`，本次只修了 FlywayMigrationIT。
