## 决策列表

### D1 — 认证架构：JwtFilter 作为唯一认证入口，SecurityConfig 不重复鉴权

**背景**：设计时 SecurityConfig `.anyExchange().authenticated()` 和 JwtFilter 双重认证，但 JwtFilter 未设置 SecurityContext，导致 Spring Security 拒绝所有合法请求。

**决策**：SecurityConfig 改为 `.anyExchange().permitAll()`，仅提供 PasswordEncoder Bean 和 CSRF 禁用。JwtFilter（`@Order(-100)`）作为唯一认证执行点：校验 Token → 查 Redis 黑名单 → 注入 `X-User-Id`/`X-Username` Header。

**替代方案**：JwtFilter 设置 `ReactiveSecurityContextHolder` + `UsernamePasswordAuthenticationToken` → 增加复杂度，M1.08 骨架阶段不必要。

**风险**：后续如需方法级 `@PreAuthorize`，需重构为 `ServerSecurityContextRepository` + `ReactiveAuthenticationManager`。

### D2 — 登出实现：Refresh Token Blacklist（Redis）

**决策**：
- Token 签发时注入 `jti`（UUID）声明
- 登出时将 Refresh Token 的 jti 写入 Redis `fin:bl:{jti}`，TTL = Token 剩余有效期
- JwtFilter 每次请求查黑名单；Redis 不可用时降级放行（`onErrorResume`）
- Access Token 不做黑名单（自然过期 1h），仅 Refresh Token 可撤销

**理由**：Access Token 无状态是 JWT 设计目标；Refresh Token 生命周期长（7d），需要撤销能力。

### D3 — 密码与账户安全

**决策**：
- BCrypt cost=10
- 登录失败统一返回 `BAD_CREDENTIALS`（不区分"用户不存在" vs "密码错误"，防用户名枚举）
- 仅禁用账户返回 `ACCOUNT_DISABLED`（已存在用户无法隐藏，且帮助排查）

**风险**：仍存在轻微用户名枚举（disabled vs bad_credentials），后续可通过 rate limiting 缓解。

### D4 — 任务状态机：21 状态 + 重试限制

**决策**：
- 完整实现 spec §3.2.1 的 21 个状态（PENDING → PARSE_* → EXTRACT_* → CHECK_* → REPORT_* → COMPLETED/FAILED/CANCELLED）
- `TaskStateMachine` 用静态 `EnumMap<TaskStatus, Set<TaskStatus>>` 定义所有合法转换（线程安全）
- M1.09 骨架 `MAX_RETRIES = 1`（失败即终止），原因：`task_step` 表无 `retry_count` 列，无法精确计数 → 无限重试风险

**TODO M2.x**：Flyway 迁移加 `task_step.retry_count` 列 → `countRetriesForStep` 精确计数 → `MAX_RETRIES` 恢复为 3 → 实现指数退避。

### D5 — 三表并行抽取：checkAllExtractsDone 空记录处理

**决策**：`checkAllExtractsDone` 对不存在的步骤记录使用 `switchIfEmpty` + MISSING 占位，确保 `all()` 在三步都实际存在时才返回 true。防止 `createStepRecords` 部分失败导致提前进 CHECK。

### D6 — R2DBC 实体：Lombok @Data + @Builder

**决策**：实体类使用 `@Data @Builder @NoArgsConstructor @AllArgsConstructor` 配合 Spring Data R2DBC 的 `@Table` / `@Id` / `@Column`。后续考虑限制 `@EqualsAndHashCode` 仅含 `@Id` 字段（避免保存前后 hashCode 变化）。

### D7 — 异常处理：R2DBC Repository 方法命名

**决策**：Spring Data R2DBC 支持从方法名推导查询（`findByUsername`、`existsByUsername`），无需手写 `@Query`。`findByUsername` 返回 `Mono<UserAccount>`（唯一约束保证最多一条）。

### D8 — TaskMessageProducer 序列化

**决策**：使用 `RabbitTemplate.convertAndSend(exchange, routingKey, Message)` 发送预制 Message 对象。当前使用独立 `ObjectMapper` 序列化，存在与 `Jackson2JsonMessageConverter`（消费者侧）不一致风险。后续统一为注入 ObjectMapper Bean。
