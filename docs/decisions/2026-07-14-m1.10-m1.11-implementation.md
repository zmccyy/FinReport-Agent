## 决策列表

### D1 — SSE 池实现：Sinks.Many.replay().limit(16)

**背景**：需在 WebFlux 环境下实现 SSE 发布/订阅，支持断线重连和多订阅者。

**决策**：使用 Project Reactor `Sinks.many().replay().limit(16)` 作为每个 taskId 的事件总线：
- `replay().limit(16)` — 新订阅者自动获取最近 16 条历史事件（断线重连支持）
- `ConcurrentHashMap<String, Sinks.Many>` — 线程安全的 sink 注册表
- `tryEmitNext` + 边界重试（8 次）处理 `FAIL_NON_SERIALIZED` 瞬态故障
- 已完成的 sink 保留在 map 中（不 remove），供重连客户端自然获取终端事件后关闭

**替代方案**：Spring SseEmitter（阻塞式，与 WebFlux 不兼容）、Redis Pub/Sub（增加复杂度，M1 不必要）

**风险**：已完成 sink 不清理会缓慢泄漏内存（每个约 1KB）。后续里程碑需增加 TTL 清理机制。

### D2 — 心跳防代理断连：takeUntilOther 自动停止

**背景**：CDN/反向代理可能因长期无数据流量断开 SSE 连接。

**决策**：每 30s 发送 SSE 注释行（`:heartbeat`）作为心跳。使用 `takeUntilOther(eventStream.then().thenReturn(true))` 让心跳在 eventStream 完成时自动停止，避免 SSE 连接泄漏。

**替代方案（已拒绝）**：`Flux.merge` 不加 takeUntilOther → 心跳 Flux.interval 永不完成，SSE 连接在任务完成后仍然保持（资源泄漏）。

### D3 — MQ 消费：手动 ack + 绕过 JSON 转换器

**背景**：spec §3.1 要求手动 ack，spec §8.3 要求业务成功才 ack。

**决策**：
- RabbitMqConfig 容器工厂设 `AcknowledgeMode.MANUAL`
- ProgressConsumer 接收原始 `Message` 参数（绕过 `Jackson2JsonMessageConverter`）手动解析 JSON
- L3（Python）生产的消息不含 `__TypeId__` 头，绕过转换器避免反序列化失败
- 处理成功 → `channel.basicAck()`；异常 → `channel.basicNack(requeue=false)` → 进 DLQ

**理由**：只有 1 个 `@RabbitListener`（ProgressConsumer），MANUAL ack 不影响其他组件。

**风险**：未来新增 `@RabbitListener` 需显式声明 `Channel` 参数 + 手动 ack，否则消息会堆积在 unacked 状态。

### D4 — JSON 序列化：统一使用 ObjectMapper

**背景**：审查发现 TaskController 用 `String.format` 构造 JSON（无转义），ProgressConsumer 用 `ObjectMapper.writeValueAsString`。

**决策**：TaskController 统一改用 `ObjectMapper` + `LinkedHashMap` 构建 SSE 事件 JSON。即使字符串来自内部（taskId），也保证 JSON 结构合法。

**替代方案**：提取共享 `SseEventBuilder` 消除两处的重复逻辑。M1 阶段暂不实施，留到 M2 重构。

### D5 — handleStepProgress 返回值利用

**背景**：审查发现 ProgressConsumer 调 `handleStepProgress().block()` 丢弃返回值，再用 `findById().block()` 二次查询。

**决策**：直接使用 `handleStepProgress` 返回的 `Task` 对象判断终态，省去一次 DB 查询。

**风险**：`prefetch=1` 保证单线程消费，无 TOCTOU 风险。若后续增加并发度，需重新评估。
