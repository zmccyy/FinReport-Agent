## 发现与处理优先级

### P0-1：开发环境无法按仓库规定的一键命令启动

- **证据**：`AGENTS.md` §7.1 指示只运行 `docker compose -f deploy/docker-compose.dev.yml up -d`；但 `deploy/docker-compose.dev.yml:2` 明确该文件应与基础文件合并使用。独立执行 `config` 已报 `service "minio" has neither an image nor a build context specified`。
- **进一步证据**：即使使用正确的双文件组合，`deploy/docker-compose.dev.yml:90-92` 覆盖 backend 命令为 `./mvnw spring-boot:run`，而 `backend/Dockerfile:54-68` 的 `dev` stage 未复制 `mvnw` 或 `.mvn/`，容器内无该可执行文件。
- **影响**：M1.02 的“docker compose up 一键启动”和后续容器联调没有可用启动路径；backend 将在 dev 容器中启动失败。
- **建议**：统一文档为双文件命令，并在 dev 镜像复制 wrapper（或改为 `mvn spring-boot:run`）。修复后从干净环境验证所有声明服务健康。

### P0-2：任务详情、进度流和取消接口没有校验任务归属

- **证据**：`backend/src/main/java/com/finreport/controller/TaskController.java:73-110`、`:121-127`、`:137-163` 仅按 taskId 查找/取消任务；没有读取 JWT filter 注入的用户 ID，也没有使用 `task.user_id` 校验。`JwtFilter.java:99-103` 仅完成认证上下文注入。
- **影响**：任意已登录用户知道或猜测 taskId 后，可以读取他人任务、订阅他人 SSE 进度，或取消他人任务。
- **建议**：在 service/repository 层提供按 `taskId + userId` 查询与取消；Controller 从可靠认证上下文读取 userId；补充跨用户 403 测试。

### P1-1：ProgressConsumer 在业务处理失败后仍 ACK，造成消息丢失

- **证据**：`backend/src/main/java/com/finreport/mq/ProgressConsumer.java:119-124` 捕获 `orchestrator.handleStepProgress(...).block()` 异常后只记录日志，随后 `:149-151` 无条件 `basicAck`。现有 `ProgressConsumerTest` 甚至将此行为锁定为 `shouldStillAckWhenOrchestratorThrows`。
- **约束冲突**：`AGENTS.md` §8.3 要求“业务成功才 ack；失败 nack(requeue=false) 进 DLQ”。
- **影响**：数据库状态推进失败时进度消息既不会重试也不会进入 DLQ，任务状态与 SSE 可能永久不一致。
- **建议**：成功处理后才 ACK；反序列化/业务失败 NACK 到 DLQ；测试应断言 nack 而非 ack。

### P1-2：Last-Event-ID 断线恢复不满足任务卡和设计要求

- **证据**：`TaskController.java:68-70` 明示参数仅用于日志；`:93-103` 始终订阅 `SseEmitterPool`。`SseEmitterPool.java:24-25,86-88` 只保存进程内最近 16 条事件；没有 Redis 持久化。`ProgressConsumer.java:186-190` 用 `taskId:step` 作为 event ID，同一步的 RUNNING/SUCCESS 会冲突。
- **影响**：不能依据客户端 Last-Event-ID 定位和补发；进程重启、事件超过 16 条、同一步多状态时均无法可靠恢复。M1.11 只能判定为部分完成。
- **建议**：将有序序列号与任务快照/事件记录持久化到 Redis，按 Last-Event-ID 返回缺失事件或最新状态；为重启、超过缓存、同步骤多事件写集成测试。

### P1-3：MD5 命中仍调度 PARSE，且缺少既有 PDF object key

- **证据**：`FileService.java:152-160` 命中 `pdf_md5` 后调用 `createTaskForReport`；`:221-235` 再调用会立即调度 PARSE 的 `orchestrator.createTask(...)`。传入 `buildPayload(...)` 不含 `report.pdfObjectKey`。相反，新文件路径在 `:200-205` 明确额外传递该 key。
- **约束冲突**：`AGENTS.md` §8.4 要求同 PDF 重传命中缓存时“跳过解析抽取”。
- **影响**：重复上传不但未复用解析结果，还会发布一个缺少 PDF 定位信息的 PARSE 消息。
- **建议**：明确缓存复用语义：复用已有完成结果时不创建解析任务；如业务上确需重新处理，至少将既有 `pdfObjectKey` 传入 payload 并命名为“重新处理”。

### P1-4：Idempotency-Key 仅“先读后写”，并发请求可重复创建任务

- **证据**：`FileService.java:118-127` 先 GET Redis，再调用上传流程；`:313-330` 只在成功结束后普通 `SET` 缓存响应，没有 claim/lock/SETNX 或数据库唯一键。
- **影响**：两个并发请求可同时读到空值并分别上传、入库、发布 MQ，违背 M1.12 的“重复 Idempotency-Key 返回原 taskId”。
- **建议**：使用 Redis 原子 `SET NX` + 处理中状态/可等待结果，或数据库唯一幂等记录；为并发双请求添加集成测试。

### P1-5：MinIO bucket 初始化脚本没有接入容器启动流程

- **证据**：`deploy/minio/init.sh` 可以创建 6 个 bucket，但 `deploy/docker-compose.yml:81-102` 仅挂载脚本为 `/data/init.sh`，命令仍是 `server /data --console-address ":9001"`，没有 init sidecar、entrypoint 包装或执行该脚本。
- **影响**：M1.04 的“容器启动后自动初始化”不成立；FileService 只会按需创建 uploads bucket，不能替代其他 5 个 bucket、生命周期和策略。
- **建议**：增加 `minio-init` one-shot service（等待 healthcheck 后执行 `mc`），并验证六个 bucket 和策略。

### P1-6：DoD 所需 80% 覆盖率没有工具强制或可复核证据

- **证据**：`backend/pom.xml:159-178` 只有 JaCoCo `prepare-agent` 和 `report`，没有 `jacoco:check`、minimum 0.80 或 CI gate；本次 verify 后仅发现 jacoco 日志文件，未发现可审计的 HTML/XML 报告。
- **影响**：即使测试通过，也不能证明 AGENTS §6.1 和 §14.1 的覆盖率 DoD 已满足。
- **建议**：修复报告生成路径/编码问题，配置 `jacoco:check` 的核心模块 80% 门槛，在 CI 发布报告。

### P2-1：M1.01 的 README 占位文件缺失

- **证据**：M1.01 任务卡将 `README.md` 列为涉及文件；仓库顶层没有该文件，`git ls-files README.md` 也为空。
- **影响**：目录已建立、Git 已初始化，但任务卡所列仓库入口缺失。
- **建议**：补充最小 README（项目用途、目录、开发/测试命令和当前状态）。

### P2-2：健康检查对 MinIO 与 AI 服务无条件报告 UP（既有风险）

- **证据**：`HealthController.java:69-79` 为 MinIO/AI 服务写死 `status=UP`；`:89-92` 因而不会将二者不可用反映到 overall status。此前 `docs/decisions/2026-07-14-max-effort-code-review-m1.md` 的 D1 已将其作为 M1 阶段的接受风险。
- **影响**：`/api/v1/system/health` 不完全符合“各组件状态”的验收语义，Docker healthcheck 可能误报应用可用。
- **建议**：在进入真实上传/AI 链路前改为有超时的真实探测，或明确拆分 liveness/readiness。

### P2-3：状态机重试次数与设计目标不一致

- **证据**：`TaskStateMachine.java:24-31` 将 `MAX_RETRIES` 固定为 1，注释承认与目标 3 次重试不同。
- **影响**：M1 骨架可以接受“失败即终止”的取舍，但不得宣称已完整满足设计状态机的重试语义。
- **建议**：在后续状态机里程碑使用 task_step retry_count 恢复三次、退避和可观测性。
