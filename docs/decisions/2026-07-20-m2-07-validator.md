## 1. 背景

M2.06 交付了 `Extractor.extract`，把 7B 模型输出解析为 `FinancialStatement`，但有两个未闭环的问题：

1. **校验链不完整**：Pydantic 在 `_coerce_to_statement` 里只做了结构校验（report_period 非空、value 非 NaN），没有校验业务规则（report_period 格式、currency ISO 4217、unit 枚举、关键科目缺失、数值范围合理性、重复科目）
2. **降级链未实现**：spec §10.3 要求"抽取小模型 JSON 失败 → 重试 1 次 (temp=0.1) → 改用 7B"，M2.06 的 `build_retry_prompt` 已准备好但没有 caller

M2.07 闭环这两点：交付 `Validator` + `extract_with_retry` + `statement_schema.json`。

## 2. 决策列表

### D1：Validator 是纯函数式校验器，不持有 model_lock 也不调 hub.generate

**选项**：
- A. Validator 内部触发重试（持有 extractor + hub + lock）
- B. Validator 只校验，重试由独立函数 `extract_with_retry` 协调

**决策**：B

**理由**：
- spec §3.9 "业务代码不直接 import transformers / vllm" 的精神是职责分离
- Validator 单一职责 = 校验，便于单测（注入 `ExtractionResult` 即可）
- 重试涉及 prompt 重构 + temp 调整 + hub.generate，是编排逻辑，不属于校验器
- M2.08 L2 编排层可能想跳过重试（如批量抽取时只校验一次），独立函数更灵活

### D2：error_hint 用 message 拼接，不用 code

**选项**：
- A. `error_hint = "; ".join(i.code for i in errors[:3])` → `"invalid_report_period; nan_value"`
- B. `error_hint = "; ".join(i.message for i in errors[:3])` → `"report_period 格式应为 YYYY-MM-DD，实际为 '2024/12/31'"`

**决策**：B

**理由**：
- spec §10.3 retry prompt 模板是 `"上次输出 JSON 解析失败（{error_hint}）"`，给模型人类可读的修正指引
- code 是机器可读的（如 `nan_value`），模型不知道怎么修正
- message 直接告诉模型"report_period 格式应为 YYYY-MM-DD"，模型能据此修正

### D3：关键科目缺失只产 warning 不阻断

**背景**：spec §2.3 M7 列了 BS 必含「资产总计」「负债合计」「所有者权益合计」等关键科目。

**选项**：
- A. 缺失关键科目 = error，触发重试
- B. 缺失关键科目 = warning，不触发重试

**决策**：B

**理由**：
- 7B 通用 prompt 偶尔会漏「所有者权益合计」（模型可能合并到「负债和所有者权益总计」）
- 强制重试会触发 temp=0.1 重新生成，但模型可能仍漏，浪费推理预算
- warning 记录到 issues，M2.08 编排层可据此决定是否人工复核或调勾稽规则兜底
- error 级只保留"绝对错误"：parse 失败 / report_period 格式错 / NaN / inf / 空 statements

### D4：currency 校验用 isascii() + isalpha() + len==3

**背景**：spec §2.3 M7 要求 currency 是 ISO 4217 三字母代码（CNY/USD/HKD）。

**陷阱**：`"人民币".isalpha()` 返回 True（中文是字母），`len("人民币") == 3`，会通过简单的 `len==3 and isalpha()` 校验。

**决策**：`len(currency) == 3 and currency.isalpha() and currency.isascii()`

**理由**：
- ISO 4217 是 ASCII 三字母代码，强制 ASCII 排除中文误判
- `isascii()` 是 Python 3.7+ 内置，无新依赖

### D5：value 上限 1e15，只产 warning

**背景**：A 股最大资产约 1 万亿元 = 1e12 元（工商银行 2023 年报资产总计 44.7 万亿元 = 4.47e13）。

**选项**：
- A. 上限 1e13（接近工行实际值）
- B. 上限 1e15（留 2 个数量级缓冲）
- C. 不设上限

**决策**：B（1e15）

**理由**：
- 1e13 会误报工行 / 农行等大行
- 1e15 留 2 个数量级缓冲，覆盖所有 A 股实际值
- 超过 1e15 几乎肯定是模型幻觉（如把"百万元"当"元"，数值放大 1e6 倍）
- warning 不阻断，避免误杀正常大额数据

### D6：statement_schema.json 手动加 $schema/$id/中文 description

**背景**：Pydantic 生成的 `model_json_schema()` 没有 `$schema` 字段，外部消费者（L2 Java / 前端 / OpenAPI 工具）需要明确的 JSON Schema 版本声明。

**决策**：
- 顶层加 `"$schema": "http://json-schema.org/draft-07/schema#"`
- 顶层加 `"$id": "https://finreport.example/schemas/statement_schema.json"`
- 用中文 description 覆盖 Pydantic 默认 description，含重新生成命令

**理由**：
- draft-07 是 JSON Schema 最稳定广泛支持的版本
- `$id` 让外部工具能稳定引用
- 中文 description 含重新生成命令，方便未来维护者同步 Pydantic 模型变更

### D7：JSON Schema drift 测试守卫

**决策**：`test_statement_schema_json_matches_pydantic_model` 比较 committed 文件与 `FinancialStatement.model_json_schema()`，strip `$schema/$id/title/description` 后必须相等。

**理由**：
- Pydantic 模型字段变更（如加 `confidence` 字段）但忘了重新生成 schema 文件 → 测试失败
- 防止 L2 Java 用过期 schema 校验 L3 输出，导致"本地通过线上失败"
- 维护成本：未来 Pydantic 模型变更时，跑一次重新生成命令即可

### D8：extract_with_retry 返回 (result, validation) 元组

**选项**：
- A. 返回 `ExtractionResult`，caller 自己再跑 `validator.validate`
- B. 返回 `tuple[ExtractionResult, ValidationResult]`

**决策**：B

**理由**：
- caller（M2.08 编排）需要知道是第一次成功还是重试成功，以决定是否记录"重试率"指标
- caller 需要 validationResult.issues 来记录 warning（如 missing_required_item）做监控
- 返回元组避免 caller 重复跑 validator，节省 CPU
- 元组比 dataclass 简单，符合 Python 习惯

## 4. 发现的风险

### R1：关键科目 warning 在 M2.08 可能被忽略

`missing_required_item` 只产 warning，如果 M2.08 编排层不消费 issues 列表，关键科目缺失会被静默忽略。

**缓解**：M2.08 实现时必须在 progress 消息里携带 `validation.issues`，让 L2 / 前端能看到 warning。M2.09 写 financial_statement 表时建议加 `validation_warnings` JSON 字段。

### R2：retry prompt 长度可能超 7B 上下文

`build_retry_prompt` 在原 prompt（含 table_html）后追加 retry 指引。如果原 prompt 已经接近 32K context limit，retry prompt 会溢出。

**缓解**：
- spec §4.7 指定 Qwen2.5-7B-Instruct 32K context
- M2.06 的 `_OUTPUT_SCHEMA_HINT` 已经很短
- 实际 table_html 通常 < 4K tokens（财报单表）
- M2.12 集成测试时验证；若溢出，在 `extract_with_retry` 加 prompt 长度检查

### R3：error_hint 拼接可能超长

如果一次校验产生 10+ error，`error_hint` 取前 3 条 message 拼接，但单条 message 可能很长（如 `"statements[balance_sheet][15].value: value 异常大（科目：应收账款，值：1e20，上限：1e15）"`）。

**缓解**：
- 当前实现取前 3 条，单条 message < 100 字符，总 hint < 300 字符，安全
- M2.12 集成测试时验证；若超长，加 `error_hint[:500]` 截断

### R4：JSON Schema drift 测试可能在 Pydantic 升级时误报

Pydantic v2.x 小版本升级可能改变 `model_json_schema()` 的输出格式（如字段顺序、`$defs` 嵌套）。

**缓解**：
- pyproject.toml 锁定 `pydantic>=2.7,<3.0`
- 测试比较时用 dict 相等（不比较字符串），字段顺序不影响
- 若 Pydantic 升级导致格式变化，重新生成 schema 文件即可
