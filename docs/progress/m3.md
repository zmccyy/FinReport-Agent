## M3.01 交付说明（2026-07-21）

### 实现文件

- `ai-service/app/schemas/reasoning.py` — 新增 schemas：
  - `RuleType` 三条硬编码勾稽规则枚举
  - `Severity` 严重度（INFO / WARN / ERROR / CRITICAL）
  - `RuleResult` 单条规则结果（对应 `accounting_check` 表）
  - `Anomaly` 异常检测结果（对应 `anomaly` 表，M3.03 填充）
  - `StatementSnapshot` 规则引擎入参，封装三表 + `require` 同义词查询
  - `CheckResult` Reasoner 整体输出（spec §2.3 M8 L213）
- `ai-service/app/modules/reasoner/accounting_rules.py` — 3 条规则实现：
  - `BalanceSheetIdentityRule` 资产=负债+权益（含 4 个权益同义词）
  - `NetIncomeToRetainedEarningsRule` 净利润→未分配利润（含盈余公积/应付股利可选调整项）
  - `CashFlowVsNetIncomeRule` 经营现金流 vs 净利润（含折旧摊销可选调整项）
- `ai-service/app/modules/reasoner/rule_engine.py` — `RuleEngine` 调度 3 条规则 + 计算 confidence + 异常降级
- `ai-service/tests/test_m3_rule_engine.py` — 20 个测试用例

### 关键设计决策

1. **Decimal 精度** — 所有数值用 `decimal.Decimal` 计算，避免 float 累积误差（spec §8.4）
2. **同义词组查询** — `StatementSnapshot.require(synonyms)` 任一命中即返回；缺失只回填代表名
3. **容差分级** — 默认绝对容差 0.01 元；规则 2/3 缺调整项时放宽到净利润的 5% / 20%
4. **缺失降级** — 必需科目缺失直接 `severity=CRITICAL`，`missing_items` 填充便于 M3.02 LLM 复核定位
5. **规则异常隔离** — `RuleEngine.check` 用 try/except 包裹单条规则，异常转 CRITICAL 不阻断其他规则
6. **confidence 计算** — 基础分 = 通过规则数 / 总数；每条 CRITICAL 额外扣 0.2；最低 0.0 最高 1.0

## M3.02 交付说明（2026-07-22）

### 实现文件

- `ai-service/app/modules/reasoner/llm_reviewer.py` — 新增 `LLMReviewer` 类 + `build_review_prompt`：
  - `LLMReviewer.review(check_result, snapshot)` 异步方法，复核 `CheckResult` 中所有 WARN/ERROR 规则
  - `build_review_prompt(rule, snapshot)` 构建 chat-style prompt（system + user + assistant 模板）
  - `_extract_json_object` 三层降级解析（直接 parse / 围栏解包 / 贪婪匹配）
  - `_append_fallback` 降级标记拼接工具
- `ai-service/app/schemas/reasoning.py` — `RuleResult` 新增 `llm_reviewed: bool = False` 字段
- `ai-service/tests/test_m3_llm_reviewer.py` — 22 个测试用例（8 个测试类）

### 关键设计决策

1. **触发条件分级** — 仅复核 `severity ∈ {WARN, ERROR}` 的规则；`INFO` 已通过无需复核；`CRITICAL` 是科目缺失/规则异常，LLM 无数据可分析。降级路径不调 LLM 节省显存。
2. **与 RuleEngine 解耦** — `RuleEngine` 保持纯同步、无 IO；`LLMReviewer` 是独立异步类。调用方（M3.04 L2 编排）按需 `await reviewer.review(...)`，未调 LLM 复核的链路零开销。
3. **同步 generate 包到 to_thread** — `ModelHub.generate` 是 torch 阻塞调用；用 `asyncio.to_thread` 避免阻塞事件循环，不引入并发（spec §8.1 单进程只能装 1 个 7B，并发无收益反而争抢）。
4. **不可变模型** — `RuleResult` / `CheckResult` 是 Pydantic 不可变模型；复核产出新对象（`model_copy(update=...)`），原对象保持不变便于 L2 落表前后对比。
5. **失败降级链（spec §10.3）** — LLM 超时 / JSON 解析失败 / 异常 / 空 reason 都不抛 `Exception`，保留原 note 并追加 `[LLM 复核失败: ...]` / `[LLM 输出无法解析为 JSON]` / `[LLM 复核超时]` / `[LLM 返回空 reason]` 标记，保证 Reasoner 链路不被 LLM 失败阻断。
6. **prompt 强约束 JSON** — 对齐 `extractor.prompts` 风格，输出 `{"reason": "...", "is_explained": true/false}`；`is_explained=false` 时 note 追加"建议人工排查"后缀。
7. **三表上下文截断** — prompt 中每张表最多 15 个科目，避免 prompt 过长撑爆 7B 上下文（spec §3.7 REASON 链路 60s SLA）。
8. **confidence 不变** — LLM 复核只解释差异，不改通过/失败状态；`CheckResult.confidence` 复核前后一致。

## M3.03 交付说明（2026-07-22）

### 实现文件

- `ai-service/app/modules/reasoner/anomaly_detector.py` — 新增 `AnomalyDetector` 类 + `AnomalyType` 枚举：
  - `AnomalyType.YOY_CHANGE` / `QOQ_CHANGE` / `LOGIC_CONFLICT` 对齐 `Anomaly.anomaly_type` 字符串值
  - `AnomalyDetector.detect(current, *, previous=None, year_ago=None)` 同步检测异常
  - `_detect_changes` 遍历三表科目计算变动比例，超阈值触发 WARN/ERROR
  - `_detect_logic_conflicts` 4 条科目逻辑冲突规则（应收/存货/净利润/OCF 与营收方向冲突）
  - `_change_ratio` 静态方法含零值保护（对比期为 0 返回 None 避免除零）
  - 5 个同义词组常量（REVENUE/AR/INVENTORY/NET_INCOME/OCF_SYNONYMS）
- `ai-service/tests/test_m3_anomaly_detector.py` — 44 个测试用例（12 个测试类）

### 关键设计决策

1. **同步纯计算** — 与 `RuleEngine` 一致，无 IO、不调 LLM；M3.02 `LLMReviewer` 可独立扩展复核逻辑异常（M3.03 不集成，避免过度设计）。
2. **多期快照** — `StatementSnapshot` 是单期；`detect` 接受 `previous` / `year_ago` 两个可选对比期，调用方（M3.04 L2 编排）负责从历史报告中查询并构造。
3. **变动异常阈值分级** — `|ratio| >= 30%` 触发 WARN，`|ratio| >= 100%` 升级 ERROR（spec §2.3 M8 L211 默认 30%）。
4. **逻辑异常 4 条规则** — 应收激增+营收下滑 / 存货激增+营收下滑 / 营收增长+净利润下滑 / 净利润增长+OCF 下滑；severity 固定 ERROR。
5. **零值保护** — 对比期值为 0 时跳过变动比例计算（除零）；本期与对比期均为 0 视为无变动。科目新增/转出（对比期无此科目）由调用方在 M3.04 编排层另行处理。
6. **科目同义词** — 复用 `StatementSnapshot.require` 按同义词组查询，容忍 A 股年报变体（"营业收入" vs "营业总收入"）。
7. **不可变输入** — `detect` 不修改输入 `StatementSnapshot`；产出 `list[Anomaly]`，调用方 `model_copy` 到 `CheckResult.anomalies`。
8. **confidence 不动** — 与 `LLMReviewer` 一致，`AnomalyDetector` 只产出 anomalies，不重新计算 `CheckResult.confidence`；M3.04 编排层可按需调整（避免双重扣分）。
9. **逻辑异常 anomaly_type 固定 LOGIC_CONFLICT** — `_build_logic_anomaly` 写入 `Anomaly.anomaly_type = "logic_conflict"`，与触发时的对比类型（YOY/QOQ）解耦；对比类型仅体现在 `description` 中（如"应收账款激增但营收下滑（同比变动：ar_ratio=50.0%, revenue_ratio=-20.0%）"）。
10. **Decimal 精度** — 变动比例用 `Decimal` 计算，避免 float 累积误差（spec §8.4 数据一致性）。

- `backend/src/main/java/com/finreport/service/orchestrator/TaskOrchestrator.java` — 注入 `CheckResultWriter`，CHECK 首次 SUCCESS 时调用 `writeCheckResult` 后再触发 REPORT
- `backend/src/test/java/com/finreport/service/reasoner/CheckResultWriterTest.java` — 23 个测试用例（3 个嵌套类：ParseCheckResult / WriteCheckResult / Transaction）

### 关键设计决策

1. **失败策略对齐 StatementWriter** — 写库失败只 log error 不抛异常，避免阻断状态机推进；REPORT 阶段会因数据缺失自然失败并暴露问题（spec §8.4 "失败不强制回滚，保留数据供排查"）。这样保持 M2.08 已稳定的 CHECK→REPORT 路径不被回滚。
2. **事务边界（B1 修复）** — `accounting_check` + `anomaly` 两张表在同一事务内顺序写入（spec §8.4 任务边界 = 事务边界）；任一表写入失败整体回滚。通过 `TransactionalOperator.transactional()` 包裹 `saveRules.flatMap(saveAnomalies)` 顺序链实现，不再用 `Mono.zip` 并行。
3. **reconcile 路径不补写（B2/B3 修复）** — 进程崩溃发生在 `step.setStatus(SUCCESS)` 之后、`writeCheckResult` 完成之前时，MQ 重放 CHECK SUCCESS 只调度 REPORT，不补写 check 结果。CHECK 链路无等价于 `ExtractCacheService` 的结果缓存，重放时拿不到原 result。此场景下 REPORT 会因数据缺失失败，由用户重传 PDF 触发整体重跑恢复（spec §8.4）。
4. **anomaly_type 白名单（M2 修复）** — `parseAnomaly` 校验 `anomaly_type ∈ {yoy_change, qoq_change, logic_conflict}`，非白名单值跳过写库并 log warn，避免脏数据污染 `anomaly` 表。
5. **payload 契约对齐 L3** — `parseCheckResult` 严格对齐 L3 `CheckResult.to_dict()` 输出格式：`rules` 是 List<Map>，每条含 `rule_type` / `rule_name` / `expected` / `actual` / `diff` / `is_pass` / `severity` / `note`；`anomalies` 同理。Decimal 字段同时支持 String 和 Number 输入（Number 用 `toString()` 中转避免 double 精度损失）。
6. **rules 缺失视为无效 payload** — CHECK 阶段至少应产出 3 条规则结果；`rules` 字段缺失、非 List、或为空列表都返回 null 跳过写库。`anomalies` 缺失视为空列表（M3.01 阶段 L3 还未集成 AnomalyDetector 时合法）。
7. **reportId 解析双路径** — 优先用 `task.ref_report_id`（M2.08 已稳定路径），缺失时回退到 `report.task_id` 关联查询。两条路径都失败时 log error 返回 0，不阻断状态机。
8. **ParsedCheckResult 用 record** — 内部解析结果容器用 Java record（AGENTS.md §3.1 DTO 用 record），不可变且自动生成 accessor。
9. **llm_reviewed 字段不落表** — `accounting_check` 表无 `llm_reviewed` 列（M3.04 决策：避免 schema 膨胀）；L2 不需要区分 note 来源，note 字段已含 LLM 复核标记（"[LLM 复核] ..."）。

### 关键设计决策

1. **5 段固定结构** — 对齐 spec §2.3 M10「公司概况 / 财务概览 / 三表分析 / 异常与风险 / 结论」。`ReportSection.section_type` 用枚举固定顺序；LLM 自定义 title 时按位置归一为 5 段固定 type（容错策略）。
2. **异步 + to_thread 包装** — `ModelHub.generate` 是 torch 阻塞调用；用 `asyncio.to_thread` 避免阻塞事件循环，不引入并发（spec §8.1 单进程只能装 1 个 7B，并发无收益反而争抢）。对齐 `LLMReviewer` 风格。
3. **不可变输入** — 不修改 `FinancialStatement` / `CheckResult`；产出新的 `ReportResult`。测试用例 `test_should_not_mutate_statement` / `test_should_not_mutate_check_result` 验证。
4. **失败降级（spec §10.3）** — LLM 超时 / 异常 / JSON 解析失败 / sections 不合规 都不抛 `Exception`，走模板降级产出 5 段齐全的 `ReportResult`（`fallback=True`），保证 Reasoner → Report 链路不被 LLM 失败阻断。`ReportResult.fallback` 字段便于 M3.09 前端展示降级提示横幅。
5. **模板降级保留 token metrics** — 失败路径仍透传 `prompt_tokens` / `completion_tokens` / `latency_ms`，便于 M3.10 SLA 度量与监控告警。
6. **prompt 强约束 JSON** — 对齐 `extractor.prompts` / `llm_reviewer` 风格，输出 `{"sections": [{"title": "...", "content": "..."}]}`；`sections` 长度必须为 5，顺序对齐固定标题。
7. **三表上下文截断** — 每表最多 15 个科目进 prompt，避免 prompt 过长撑爆 7B 上下文（spec §3.7 REPORT 45s SLA）。
8. **KB 检索段落** — 当前 M3 阶段知识库未建好（M5），`kb_snippets` 支持 None 或空列表；M5 落地后由调用方填充。每条段落 ≤ 500 字符、最多 5 条，避免 prompt 过长。
9. **温度 0.3** — 报告需自然语言，温度略高避免死板；但仍偏低避免幻觉（spec §10.3 复核用 0.1，报告用 0.3）。
10. **max_new_tokens = 2048** — 5 段报告较长，需要较大 token 预算（M3.02 复核 512 / M3.05 报告 2048）。
11. **timeout 45s** — 对齐 spec §3.7 REPORT 链路 SLA 目标 45s（不是超时上限 90s）。超时上限由 L2 状态机层面的 retry 计数器处理。
12. **decimal.Decimal 精度** — 数值字段统一 `Decimal(str(float))` 避免 IEEE754 精度损失（spec §8.4 数据一致性）。与 `accounting_rules.py` / `anomaly_detector.py` / `llm_reviewer.py` 风格一致。

4. **不可变输入** — 不修改 `ReportResult` / `ChartResult`；产出新的 `PdfResult`。与 `ChartRenderer` / `ReportGenerator` 风格一致。
5. **失败降级占位 PDF（spec §10.3）** — WeasyPrint 异常 / 模板缺失 / markdown 解析失败 / 超时 都不抛 `Exception`，返回最小占位 PDF（`fallback=True`），保证 Report → Artifact 链路不被 PDF 渲染失败阻断。降级链：`convert` 异常 → `_fallback()` → `_render_fallback_pdf()` → WeasyPrint 再次失败 → `_MINIMAL_PDF_PLACEHOLDER`（11 字节 PDF header `b"%PDF-1.4\n%%EOF\n"`）。
6. **异步 + to_thread + wait_for** — WeasyPrint `write_pdf` 是同步阻塞调用（内部 C 库 libpango）；用 `asyncio.to_thread` 避免阻塞事件循环，`asyncio.wait_for` 应用 `timeout_seconds`（spec §3.7 REPORT 链路 45s SLA；PDF 渲染预留 30s）。超时后底层线程仍可能继续运行（Python asyncio 无法强制取消同步线程），但调用方不会无限等待。
7. **charts 入口截断** — spec §2.3 M10 固定 3 张图表；schema `chart_count le=3` 自我保护，多余的图表在 `convert` 入口截断（避免调用方手动裁剪 + schema 校验失败）。
8. **page_size 传入模板** — `PdfConverter.page_size` 属性通过 `template.render(page_size=self.page_size)` 传入模板；`@page size: {{ page_size }}` 动态渲染，避免接口契约不一致（属性可配但模板硬编码）。
9. **markdown 扩展** — `_MARKDOWN_EXTENSIONS = ["tables", "fenced_code", "nl2br"]` 支持表格 / 围栏代码块 / 软换行转 `<br>`；与 `ReportResult.to_markdown()` 输出格式对齐。
10. **timeout_seconds 默认 30s** — spec §3.7 REPORT 链路 SLA 目标 45s（含 LLM 生成 + PDF 渲染）；PDF 渲染预留 30s，给 LLM 留 15s 余量。

### 关键设计决策

1. **失败策略（spec §8.4）** — 单文件上传失败只 log warn + 标记 `status=FAILED`，保留行供排查；不阻断后续文件上传；不抛异常避免阻断状态机推进。Report → COMPLETED 链路即使产物部分失败也能完成；前端 ReportViewer 通过 (reportId, artifactType) 查询下载链接时遇到 `status=FAILED` 会展示「产物不可用」提示。
2. **`onErrorReturn(false)` vs `onErrorResume(Mono.empty())`** — reactor 关键陷阱：`onErrorResume(e -> Mono.empty())` 会让 Mono 进入 empty 状态，后续 `.map(success -> ...)` 不执行，整个 chain 断开。正确做法是 `.onErrorReturn(false).map(success -> { if (!success) markFailed(...) })`，让 Mono 继续传递 `false` 而非进入 empty。
3. **object key 命名（spec §5.3）** — `reports/{reportId}/report.pdf` / `reports/{reportId}/report.md` / `reports/{reportId}/charts/chart_{pie|line|bar}.png`；图表文件名按 artifact_type 映射，与 L3 ChartType 枚举解耦，便于前端按固定路径拼接预签名 URL。
4. **MinIO 受检异常处理** — minio 8.5.10 的 `PutObjectArgs.contentType()` 在 builder 链中声明 `throws NoSuchAlgorithmException, IOException, KeyManagementException`；在 lambda（如 `ArgumentMatchers.argThat()`）内部调用会编译失败。解决方案：测试中改用 `args.object()` 匹配路径（如 `args.object().endsWith("report.pdf")`），避免在 lambda 内调用受检异常方法。
5. **幂等性** — `(reportId, artifactType)` 唯一性检查；重放 REPORT SUCCESS 时已存在 GENERATED 行则跳过（避免 MinIO 重复上传 + 表数据膨胀），FAILED 行允许覆盖（重试场景）。`TaskOrchestrator.handleStepSuccess` 已对重放 SUCCESS 做去重（reconcile 路径不调本类），双层保险。
6. **事务边界（spec §8.4）** — 所有 `report_artifact` 行在同一 `TransactionalOperator.transactional()` 内顺序写入，任一行写入失败整体回滚避免半成品数据。MinIO 上传在事务外执行（MinIO 失败不强制回滚，保留半成品 object 供排查）。
7. **顺序上传而非并行** — 5 个文件规模下顺序执行毫秒级完成，事务内并行 save 可能产生连接池压力且无显著收益。
8. **boundedElastic 调度器** — MinIO putObject 是阻塞 IO 调用，订阅在 `Schedulers.boundedElastic()` 上避免阻塞 Reactor 事件循环（spec §12.2 性能禁忌）。
9. **bucket 自动创建** — `ensureBucketExists` 容错处理 `BucketAlreadyOwnedByYou`（并发场景两个请求同时 makeBucket），其他 ErrorResponseException 抛出。
10. **reportId 解析回退** — 优先用 `task.ref_report_id`（M3.07 后端在创建 report 后回写），缺失时回退到 `report.task_id` 关联查询；与 `CheckResultWriter.resolveReportId` 一致。

### 关键设计决策

1. **大小写归一化而非改 L3 序列化** — 改 L3 `ChartType` 用 `use_enum_values=False` 或自定义 `model_dump` 会破坏 L3 内部一致性；L2 入口归一化更稳健，且未来 L9 Agent 编排可能产生任意大小写输入，统一兜底。

### 关键设计决策

1. **Mock L3 + 真实 L2 持久化层** — 测试用 Testcontainers MySQL + Redis（真实 R2DBC + 事务 + 缓存），Mock RabbitMQ 与 MinIO（外部依赖）。这样既验证 L2 编排层自身的端到端延迟（毫秒级，远低于 SLA），又避免 GPU 推理环境依赖。真实 L3 推理 SLA 由独立脚本 `scripts/eval_m2_sla.py` 在 GPU 环境运行，不在 CI 中执行（spec §12.1 注释）。
2. **SLA 阈值常量永久固化** — 把 spec §3.7 的 5 个 SLA 阈值作为 `private static final Duration` 常量写入测试类，未来误改阈值会编译失败或测试失败，作为「契约钉子」防止 SLA 漂移。
3. **`task_step.duration_ms` 写入路径回归** — SLA 监控依赖该字段（spec §3.7）。本次测试把它作为回归点固化：每阶段 SUCCESS 后断言 `durationMs != null && >= 0`，防止 future 重构 `TaskOrchestrator.handleStepSuccess` 破坏写入路径。
4. **MySQL DATETIME 精度修复 vs JVM 计算修复** — 两个修复路径均可：
   - 选项 A（采用）：DB 升级 `DATETIME(3)` 对齐 JVM 毫秒精度
   - 选项 B（否决）：JVM 读回 `startedAt` 后用 `Duration.between(startedAt, finishedAt)` 而非 `Duration.between(startedAt, now)` 计算时长
   
   选 A 的理由：spec §3.7 SLA 监控查询（"最近 1 小时各阶段 P95 耗时"）直接查 `started_at / finished_at` 列；如果 DB 列只有秒级精度，监控聚合结果会丢毫秒信息，多任务并发时 P95 抖动严重。升级 DB 精度一次性解决监控 + 计算两个问题，且 `DATETIME(3)` 与 `LocalDateTime.now()` 默认精度天然对齐。
5. **`@TestMethodOrder` 而非测试数据隔离** — V6 测试 `shouldEnforceUserScopedDedupeAndAllowReportTaskReuse` 先插入 `(userId=1, pdf_md5=PDF_MD5)`，V7 测试 `shouldExpandRuleTypeToVarchar32...` 通过 `ensureReportForRuleTypeCheck` 复用同一条 report。如果 V7 先运行，V6 会因唯一约束 `uk_report_user_md5` 冲突而失败。修复方式两选一：
   - 选项 A（采用）：`@TestMethodOrder(MethodOrderer.MethodName.class)` 保证 V6 字母序在 V7 之前
   - 选项 B（否决）：V7 用独立 `pdf_md5` / `userId` 插入新 report
   
   选 A 的理由：V7 测试本来就是「复用 V6 已插入的 report 验证 rule_type 扩容」，强行用独立 pdf_md5 反而破坏测试间的语义关系。`@TestMethodOrder` 显式声明依赖顺序，更符合测试作为「迁移历史 + 数据完整性」回归套件的本质。
6. **V8 迁移注释引用 MySQL 官方文档** — 在 V8 SQL 注释中明确引用 MySQL 文档原文「fractional seconds are rounded, not truncated」，把根因证据永久留在迁移文件中，避免未来误删 V8 或回退到 `DATETIME(0)`。
