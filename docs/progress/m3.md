## M3.01 交付说明（2026-07-21）

### 实现文件

- `ai-service/app/schemas/reasoning.py` — 新增 schemas：
  - `RuleType` 三条硬编码勾稽规则枚举
  - `Severity` 严重度（INFO / WARN / ERROR / CRITICAL）
  - `RuleResult` 单条规则结果（对应 `accounting_check` 表）
  - `Anomaly` 异常检测结果（对应 `anomaly` 表，M3.03 填充）
  - `StatementSnapshot` 规则引擎入参，封装三表 + `require` 同义词查询
  - `CheckResult` Reasoner 整体输出（spec §2.3 M8 L213）
- `ai-service/app/modules/reasoner/accounting_rules.py` — 3 条规则实现：
  - `BalanceSheetIdentityRule` 资产=负债+权益（含 4 个权益同义词）
  - `NetIncomeToRetainedEarningsRule` 净利润→未分配利润（含盈余公积/应付股利可选调整项）
  - `CashFlowVsNetIncomeRule` 经营现金流 vs 净利润（含折旧摊销可选调整项）
- `ai-service/app/modules/reasoner/rule_engine.py` — `RuleEngine` 调度 3 条规则 + 计算 confidence + 异常降级
- `ai-service/tests/test_m3_rule_engine.py` — 20 个测试用例

### 关键设计决策

1. **Decimal 精度** — 所有数值用 `decimal.Decimal` 计算，避免 float 累积误差（spec §8.4）
2. **同义词组查询** — `StatementSnapshot.require(synonyms)` 任一命中即返回；缺失只回填代表名
3. **容差分级** — 默认绝对容差 0.01 元；规则 2/3 缺调整项时放宽到净利润的 5% / 20%
4. **缺失降级** — 必需科目缺失直接 `severity=CRITICAL`，`missing_items` 填充便于 M3.02 LLM 复核定位
5. **规则异常隔离** — `RuleEngine.check` 用 try/except 包裹单条规则，异常转 CRITICAL 不阻断其他规则
6. **confidence 计算** — 基础分 = 通过规则数 / 总数；每条 CRITICAL 额外扣 0.2；最低 0.0 最高 1.0

## M3.02 交付说明（2026-07-22）

### 实现文件

- `ai-service/app/modules/reasoner/llm_reviewer.py` — 新增 `LLMReviewer` 类 + `build_review_prompt`：
  - `LLMReviewer.review(check_result, snapshot)` 异步方法，复核 `CheckResult` 中所有 WARN/ERROR 规则
  - `build_review_prompt(rule, snapshot)` 构建 chat-style prompt（system + user + assistant 模板）
  - `_extract_json_object` 三层降级解析（直接 parse / 围栏解包 / 贪婪匹配）
  - `_append_fallback` 降级标记拼接工具
- `ai-service/app/schemas/reasoning.py` — `RuleResult` 新增 `llm_reviewed: bool = False` 字段
- `ai-service/tests/test_m3_llm_reviewer.py` — 22 个测试用例（8 个测试类）

### 关键设计决策

1. **触发条件分级** — 仅复核 `severity ∈ {WARN, ERROR}` 的规则；`INFO` 已通过无需复核；`CRITICAL` 是科目缺失/规则异常，LLM 无数据可分析。降级路径不调 LLM 节省显存。
2. **与 RuleEngine 解耦** — `RuleEngine` 保持纯同步、无 IO；`LLMReviewer` 是独立异步类。调用方（M3.04 L2 编排）按需 `await reviewer.review(...)`，未调 LLM 复核的链路零开销。
3. **同步 generate 包到 to_thread** — `ModelHub.generate` 是 torch 阻塞调用；用 `asyncio.to_thread` 避免阻塞事件循环，不引入并发（spec §8.1 单进程只能装 1 个 7B，并发无收益反而争抢）。
4. **不可变模型** — `RuleResult` / `CheckResult` 是 Pydantic 不可变模型；复核产出新对象（`model_copy(update=...)`），原对象保持不变便于 L2 落表前后对比。
5. **失败降级链（spec §10.3）** — LLM 超时 / JSON 解析失败 / 异常 / 空 reason 都不抛 `Exception`，保留原 note 并追加 `[LLM 复核失败: ...]` / `[LLM 输出无法解析为 JSON]` / `[LLM 复核超时]` / `[LLM 返回空 reason]` 标记，保证 Reasoner 链路不被 LLM 失败阻断。
6. **prompt 强约束 JSON** — 对齐 `extractor.prompts` 风格，输出 `{"reason": "...", "is_explained": true/false}`；`is_explained=false` 时 note 追加"建议人工排查"后缀。
7. **三表上下文截断** — prompt 中每张表最多 15 个科目，避免 prompt 过长撑爆 7B 上下文（spec §3.7 REASON 链路 60s SLA）。
8. **confidence 不变** — LLM 复核只解释差异，不改通过/失败状态；`CheckResult.confidence` 复核前后一致。

## M3.03 交付说明（2026-07-22）

### 实现文件

- `ai-service/app/modules/reasoner/anomaly_detector.py` — 新增 `AnomalyDetector` 类 + `AnomalyType` 枚举：
  - `AnomalyType.YOY_CHANGE` / `QOQ_CHANGE` / `LOGIC_CONFLICT` 对齐 `Anomaly.anomaly_type` 字符串值
  - `AnomalyDetector.detect(current, *, previous=None, year_ago=None)` 同步检测异常
  - `_detect_changes` 遍历三表科目计算变动比例，超阈值触发 WARN/ERROR
  - `_detect_logic_conflicts` 4 条科目逻辑冲突规则（应收/存货/净利润/OCF 与营收方向冲突）
  - `_change_ratio` 静态方法含零值保护（对比期为 0 返回 None 避免除零）
  - 5 个同义词组常量（REVENUE/AR/INVENTORY/NET_INCOME/OCF_SYNONYMS）
- `ai-service/tests/test_m3_anomaly_detector.py` — 44 个测试用例（12 个测试类）

### 关键设计决策

1. **同步纯计算** — 与 `RuleEngine` 一致，无 IO、不调 LLM；M3.02 `LLMReviewer` 可独立扩展复核逻辑异常（M3.03 不集成，避免过度设计）。
2. **多期快照** — `StatementSnapshot` 是单期；`detect` 接受 `previous` / `year_ago` 两个可选对比期，调用方（M3.04 L2 编排）负责从历史报告中查询并构造。
3. **变动异常阈值分级** — `|ratio| >= 30%` 触发 WARN，`|ratio| >= 100%` 升级 ERROR（spec §2.3 M8 L211 默认 30%）。
4. **逻辑异常 4 条规则** — 应收激增+营收下滑 / 存货激增+营收下滑 / 营收增长+净利润下滑 / 净利润增长+OCF 下滑；severity 固定 ERROR。
5. **零值保护** — 对比期值为 0 时跳过变动比例计算（除零）；本期与对比期均为 0 视为无变动。科目新增/转出（对比期无此科目）由调用方在 M3.04 编排层另行处理。
6. **科目同义词** — 复用 `StatementSnapshot.require` 按同义词组查询，容忍 A 股年报变体（"营业收入" vs "营业总收入"）。
7. **不可变输入** — `detect` 不修改输入 `StatementSnapshot`；产出 `list[Anomaly]`，调用方 `model_copy` 到 `CheckResult.anomalies`。
8. **confidence 不动** — 与 `LLMReviewer` 一致，`AnomalyDetector` 只产出 anomalies，不重新计算 `CheckResult.confidence`；M3.04 编排层可按需调整（避免双重扣分）。
9. **逻辑异常 anomaly_type 固定 LOGIC_CONFLICT** — `_build_logic_anomaly` 写入 `Anomaly.anomaly_type = "logic_conflict"`，与触发时的对比类型（YOY/QOQ）解耦；对比类型仅体现在 `description` 中（如"应收账款激增但营收下滑（同比变动：ar_ratio=50.0%, revenue_ratio=-20.0%）"）。
10. **Decimal 精度** — 变动比例用 `Decimal` 计算，避免 float 累积误差（spec §8.4 数据一致性）。

## M3.04 交付说明（2026-07-22）

### 实现文件

- `backend/src/main/java/com/finreport/domain/entity/AccountingCheck.java` — 映射 `accounting_check` 表（V2__init_report.sql §2.3），Lombok @Data @Builder
- `backend/src/main/java/com/finreport/domain/entity/AnomalyRecord.java` — 映射 `anomaly` 表（V2__init_report.sql §2.4），类名用 `AnomalyRecord` 避免与 L3 schema 概念混淆
- `backend/src/main/java/com/finreport/repository/AccountingCheckRepository.java` — `ReactiveCrudRepository`，含 `countByReportId` / `findByReportIdOrderByRuleTypeAsc` / `deleteByReportId`
- `backend/src/main/java/com/finreport/repository/AnomalyRepository.java` — `ReactiveCrudRepository`，含 `countByReportId` / `findByReportIdOrderByAnomalyTypeAscItemNameAsc` / `deleteByReportId`
- `backend/src/main/java/com/finreport/service/reasoner/CheckResultWriter.java` — 消费 L3 CHECK progress payload 写入 `accounting_check` + `anomaly` 表
- `backend/src/main/java/com/finreport/service/orchestrator/TaskOrchestrator.java` — 注入 `CheckResultWriter`，CHECK 首次 SUCCESS 时调用 `writeCheckResult` 后再触发 REPORT
- `backend/src/test/java/com/finreport/service/reasoner/CheckResultWriterTest.java` — 23 个测试用例（3 个嵌套类：ParseCheckResult / WriteCheckResult / Transaction）

### 关键设计决策

1. **失败策略对齐 StatementWriter** — 写库失败只 log error 不抛异常，避免阻断状态机推进；REPORT 阶段会因数据缺失自然失败并暴露问题（spec §8.4 "失败不强制回滚，保留数据供排查"）。这样保持 M2.08 已稳定的 CHECK→REPORT 路径不被回滚。
2. **事务边界（B1 修复）** — `accounting_check` + `anomaly` 两张表在同一事务内顺序写入（spec §8.4 任务边界 = 事务边界）；任一表写入失败整体回滚。通过 `TransactionalOperator.transactional()` 包裹 `saveRules.flatMap(saveAnomalies)` 顺序链实现，不再用 `Mono.zip` 并行。
3. **reconcile 路径不补写（B2/B3 修复）** — 进程崩溃发生在 `step.setStatus(SUCCESS)` 之后、`writeCheckResult` 完成之前时，MQ 重放 CHECK SUCCESS 只调度 REPORT，不补写 check 结果。CHECK 链路无等价于 `ExtractCacheService` 的结果缓存，重放时拿不到原 result。此场景下 REPORT 会因数据缺失失败，由用户重传 PDF 触发整体重跑恢复（spec §8.4）。
4. **anomaly_type 白名单（M2 修复）** — `parseAnomaly` 校验 `anomaly_type ∈ {yoy_change, qoq_change, logic_conflict}`，非白名单值跳过写库并 log warn，避免脏数据污染 `anomaly` 表。
5. **payload 契约对齐 L3** — `parseCheckResult` 严格对齐 L3 `CheckResult.to_dict()` 输出格式：`rules` 是 List<Map>，每条含 `rule_type` / `rule_name` / `expected` / `actual` / `diff` / `is_pass` / `severity` / `note`；`anomalies` 同理。Decimal 字段同时支持 String 和 Number 输入（Number 用 `toString()` 中转避免 double 精度损失）。
6. **rules 缺失视为无效 payload** — CHECK 阶段至少应产出 3 条规则结果；`rules` 字段缺失、非 List、或为空列表都返回 null 跳过写库。`anomalies` 缺失视为空列表（M3.01 阶段 L3 还未集成 AnomalyDetector 时合法）。
7. **reportId 解析双路径** — 优先用 `task.ref_report_id`（M2.08 已稳定路径），缺失时回退到 `report.task_id` 关联查询。两条路径都失败时 log error 返回 0，不阻断状态机。
8. **ParsedCheckResult 用 record** — 内部解析结果容器用 Java record（AGENTS.md §3.1 DTO 用 record），不可变且自动生成 accessor。
9. **llm_reviewed 字段不落表** — `accounting_check` 表无 `llm_reviewed` 列（M3.04 决策：避免 schema 膨胀）；L2 不需要区分 note 来源，note 字段已含 LLM 复核标记（"[LLM 复核] ..."）。
