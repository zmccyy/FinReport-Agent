### 设计决策
- **单模型驻存 + key-based no-op**：6GB VRAM 只能装 1 个 7B(4bit ~5GB) 或 1 个 1.5B + 1 个 small（spec §8.1）。`LlmLoader` 用 `_loaded_key` 跟踪逻辑模型名，重复 load 同 key 直接 no-op；切 key 先 `unload()` 释放显存再 load 新模型
- **GPTQ-Int4 让 transformers 自动检测**：Qwen2.5-7B-Instruct-GPTQ-Int4 的 `config.json` 已含 GPTQ 量化配置，`_build_quant_config` 返回 `None` 让 transformers 自动应用；NF4 才显式构造 `BitsAndBytesConfig(load_in_4bit=True, bnb_4bit_quant_type="nf4", bnb_4bit_compute_dtype=torch.bfloat16, bnb_4bit_use_double_quant=True)`
- **懒加载 torch/transformers**：`TransformersBackend.load()` 内部用 `_lazy_import_torch()` / `_lazy_import_transformers()` 导入，模块顶层无重型依赖；导入失败映射为 `ModelLoadException`。单测通过 `monkeypatch.setitem(sys.modules, ...)` 注入 stub，无需 GPU/真实模型
- **推理超时用 daemon 线程**：不用 `signal.SIGALRM`（Windows 不支持）；`generate()` 启 daemon 线程跑 `model.generate()`，主线程 `thread.join(timeout=...)`，超时后 `_reset()` 释放模型抛 `InferenceTimeoutException`（spec §10.3 降级链：本地 7B 超时 → 切 API 72B）
- **OOM 启发式检测**：`_is_oom(error)` 通过错误消息 `"out of memory" / "cuda oom"` 匹配，映射为 `ModelLoadException` 触发 spec §10.3 的"释放模型 → 重启 ModelHub → 重试 1 次 → 转 API"降级链
- **embed() stub**：M2.04 只实现 7B REASON 路径；`embed()` / EMBED 场景 / LAYOUT 场景抛 `AiException`，留给 M5（bge-small-zh LoRA）和 M4（LayoutLMv3）

### 设计决策
- **不引入 fakeredis**：AGENTS.md §9.2 禁止新依赖；测试文件内置 `_FakeRedis` 实现 `set`/`get`/`delete`/`expire`/`eval`/`ping` 最小集 + 内嵌 Lua 解释器（release/refresh）
- **非重入锁 + Lua owner 校验**：`SET key worker_id NX EX ttl` 原子获取；release/refresh 用 Lua 脚本 `if GET key == ARGV[1] then DEL/EXPIRE end` 防止 worker 释放已过期 + 被另一 worker 重新获取的锁（spec §3.9 spec canonical 模式）
- **load_for_scene_with_lock 不自动 release**：返回 `ModelLock` 让调用方控制推理窗口的生命周期；M2.06+ 的 Extractor handler 在 `with lock: model.generate(...)` 块里持有，结束时自动 release。load 本身失败时 `try/except` 自动 release 兜底
- **LRU evict_idle 只卸载驻存模型**：`_last_used` 里同模型仍 resident 时才 `hub.unload()`；非驻存模型的 stale entry 静默 pop。M2.05 不接后台 tick，留给 M3/M5 的定时任务或 MQ 空闲槽位触发
- **RedisClient 接受 RedisLike 而非 redis.Redis**：`VramScheduler(redis_client=...)` 同时接受 `RedisClient` 包装（自动 unwrap）或裸 Redis-like 客户端（测试直接注入 `_FakeRedis`）；上层 `RedisClient.ping` 失败转 False 不抛，避免 health endpoint 抛 500
- **未集成 MQ consumer**：M2.05 只交付核心组件 + 单测，不动 `app/mq/consumer.py`。M2.06+ 的 extract handler 在消费 `q.extract.requests` 时显式 `VramScheduler.load_for_scene_with_lock(Scene.EXTRACT)`，busy 时抛 `ModelLockBusyException` → handler 转 `nack(requeue=True)` + sleep `model_lock_retry_seconds`（spec §3.9 "nack(requeue=true, delay=2s)"）。这样保持 M1.14 已稳定的 MQ 链路不被回滚

### 设计决策
- **Extractor 不持有 model_lock**：spec §3.9 要求"业务代码不直接 import transformers / vllm"；同理 Extractor 不该耦合显存调度。`Extractor.extract` 只调 `hub.generate()`，caller（M2.08 L2 编排或 M2.07+ L3 handler）负责 `with VramScheduler.load_for_scene_with_lock(Scene.EXTRACT) as lock: extractor.extract(...)`。这样 Extractor 可单测（注入 `_StubHub`）且与 M2.05 lock 解耦
- **失败不抛异常返回 `ExtractionResult(success=False, error=...)`**：spec §10.3 降级链要求"抽取小模型 JSON 失败 → 重试 1 次 (temp=0.1) → 改用 7B"。M2.06 把 JSON parse / schema 校验失败的产物连同 `raw_text` 一起返回，留给 M2.07 validator 用 `build_retry_prompt(error_hint=...)` 重试，再失败才上抛。这样 M2.06 单测能覆盖所有失败路径而不被异常打断
- **3 级 JSON 解析回退**：Qwen2.5-Instruct 在低温度下大部分直接输出 JSON，但偶有 ```json fenced block 或 prose 嵌入（"以下是抽取结果：{...}"）。`_extract_json_object` 依次尝试 `json.loads(text)` → `re.search(r"```(?:json)?\s*(\{.*?\})\s*```", text, re.DOTALL)` → `re.search(r"\{.*\}", text, re.DOTALL)`，三级回退确保 spec §M7 的 JSON 解析率 > 90% 验收标准
- **`_coerce_to_statement` 接受 `.value` 或 `.name` 作为 statements key**：模型输出 `{"statements": {"balance_sheet": [...]}}` 或 `{"statements": {"BALANCE_SHEET": [...]}}` 都接受；同时过滤掉非 expected_type 的科目（避免 BS prompt 混入 IS 项）。`StatementItem.value` 拒绝 NaN（spec §2.3 M7 数值合法性），千分位字符串 `"1,234,567"` 显式拒绝并返回失败（留给 M2.07 重试或上层转换）
- **`build_extract_prompt` 用 `<|im_start|>` chat 格式**：spec §4.7 指定 Qwen2.5-Instruct 系列；prompt 末尾 `<|im_start|>assistant\n` 让模型直接续写 JSON 而不复述指令。`_OUTPUT_SCHEMA_HINT` 用 `<statement_type>` 占位符 + `.replace()` 注入 `statement_type.value`，避免 f-string 嵌套转义
- **`build_retry_prompt(original_prompt, *, error_hint)`**：M2.07 validator 检测到 JSON 解析失败后，把原 prompt + 错误提示拼成 retry prompt，temp 降到 0.1 重试一次（spec §10.3 降级链）。retry prompt 在原 prompt 后追加 `"上次输出存在以下问题：{error_hint}，请修正后重新输出严格 JSON。"` 不破坏 system role
- **`FinancialStatement.to_dict()` 用字符串 key**：Pydantic v2 `model_dump()` 默认用 enum key，但下游 JSON 序列化（FastAPI / RabbitMQ）要求字符串 key；`to_dict()` 显式 `statement_type.value` 转 key，与 `_OUTPUT_SCHEMA_HINT` 输出格式对称
- **万元单位 + 负值显式测试**：A 股财报常见"单位：万元"且负值（如净利润 -1,234.56 万元）；`test_extract_handles_wan_yuan_unit` / `test_extract_handles_negative_values` 确保数值合法性 + 单位字段透传

1. 第一次抽取（默认 temp）

### 设计决策
- **Validator 不持有 model_lock 也不调 hub.generate**：spec §3.9 要求"业务代码不直接 import transformers / vllm"；同理 Validator 不耦合显存调度。`Validator.validate` 是纯函数式校验，重试由 `extract_with_retry` 函数通过 `Extractor.extract_with_prompt` 公开方法走
- **error_hint 取前 3 条 error message 拼接**：spec §10.3 要求 retry prompt 含"上次输出 JSON 解析失败（{hint}）"；error_hint 是 message 而非 code，给模型人类可读的修正指引（如 `"report_period 格式应为 YYYY-MM-DD，实际为 '2024/12/31'"`）
- **关键科目缺失只产 warning 不阻断**：spec §2.3 M7 列了"必填科目"但没要求硬阻断；模型偶尔会漏「所有者权益合计」，warning 记录即可，M2.08 编排层可据此决定是否人工复核。error 级只保留 parse 失败 / report_period 格式 / NaN/inf / 空 statements 这种"绝对错误"
- **currency 校验用 isascii() + isalpha()**：避免「人民币」这类 3 字符中文通过 ISO 4217 校验；spec §2.3 M7 要求 currency 是 ISO 4217 三字母代码（CNY/USD/HKD 等 ASCII）
- **value 上限 1e15**：A 股最大资产 1 万亿元 = 1e12 元；超过 1e15 几乎肯定是模型幻觉（如把"百万元"当"元"），warning 提示但不阻断
- **statement_schema.json 加 $schema/$id/中文 description**：Pydantic 生成的 schema 没有 $schema 字段；手动加 draft-07 声明 + 中文 description（含重新生成命令），让外部消费者（L2 / 前端 / OpenAPI）能引用
- **JSON Schema drift 测试**：`test_statement_schema_json_matches_pydantic_model` 比较文件与 `FinancialStatement.model_json_schema()`，strip `$schema/$id/title/description` 后必须相等。如果未来 Pydantic 模型字段变更但忘了重新生成 schema 文件，测试会失败

### 设计决策
- **Redis AtomicInteger 热路径 + MySQL fallback**：spec §3.2.1 任务状态机要求 3 表都 SUCCESS 才触发 CHECK；Redis `INCR` 原子计数 O(1)，避免每次 progress 都全表扫 `task_step`。Redis 故障时 fallback 到 `checkAllExtractsDone`（MySQL 全表扫描 + `Flux.all()` 短路语义）保证可用性
- **tracker.reset 在 dispatchAll 入口**：重试场景下上一轮 Redis 残留计数会污染本轮；dispatchAll 第一步 `reset(taskId)` 清空 done count + failed flag，保证幂等
- **dispatchAll 用 concatMap 顺序发布**：虽然 3 条消息逻辑上"并行"（L3 端 3 个 worker 同时消费），但 L2 发布用 `concatMap` 保证 BS → IS → CF 顺序写入 MQ，便于日志追踪；下游 3 个 worker 并发消费实现"三表并行"
- **失败 flag 阻断 CHECK**：任意 1 条 extract 失败时 `recordFailure` 写 `fin:extract:failed:{taskId} = 1`，后续 sibling SUCCESS 即使 count >= 3 也被 `hasFailed` 阻断；避免迟到的 SUCCESS 触发 CHECK 进入无意义勾稽
- **terminal FAILED 保留 failed flag**：重试耗尽转 terminal FAILED 时不清除 failed flag，防止 redelivered sibling SUCCESS 触发 CHECK；flag TTL=1d 自动过期
- **reconcile 重放路径绕过 Redis tracker**：MQ redeliver 场景下，L2 收到重复 SUCCESS 但无法重建 `recordSuccess` 调用（已过 tracker.reset 窗口）；reconcile 路径直接走 MySQL `checkAllExtractsDone` 保持真相源
- **dispatchSingle（EXTRACT_RETRY 路径）clearFailure**：单步重试时清除失败 flag，避免上轮失败残留阻断本轮 CHECK；不影响其他 sibling 步骤的 done count
- **Mockito strict stubbing 配对 stub**：`ExtractDispatcherTest.shouldMarkTaskAndStepFailedWhenMqPublishFails` 中 dispatchAll 用 concatMap 顺序发布 BS → IS → CF；strict stubbing 模式下 IS 的 `doThrow` 必须配对 BS 的 `doNothing`，否则 BS 调用时 Mockito 报 PotentialStubbingProblem
- **TaskOrchestratorTest CF stub 用 lenient**：`checkAllExtractsDone` 用 `Flux.all()` 短路语义，BS SUCCESS + IS MISSING → predicate false → CF stub 未被订阅；用 `lenient()` 避免 UnnecessaryStubbing
- **JaCoCo offline instrumentation 模式**：用户路径 `Zm'CC'y'y` 含单引号 + 工作区路径 `E:\项目\...` 含非 ASCII，surefire `CommandLineUtils.translateCommandline` 把单引号当引号字符 + Boot Manifest-JAR classpath 在 GBK JNU 解码下崩溃；切换到 `jacoco:instrument` 编译时插桩 + `forkCount=0` 在 Maven JVM 内跑测试，完全绕开 `-javaagent` 注入路径

### 设计决策
- **跨用户共享缓存**：spec §3.10 明确"同 PDF 内容（同 md5）的 extract 结果对任意用户都成立"；缓存 key 不含 userId，相同 pdf_md5 命中同一份缓存，最大化缓存命中率
- **键设计 `fin:cache:extract:{pdfMd5}:{step.name()}`**：与 `fin:extract:done:{taskId}` 计数器、`fin:extract:failed:{taskId}` 失败 flag 同前缀家族，便于运维批量扫描；按 step 分 key 而非整 JSON 一份，让单步重试 cache miss 时其他步仍可命中（虽然当前 `lookupAll` 仍要求全命中才 replay，但留扩展空间）
- **TTL=7d**：spec §3.10 锁定；7 天覆盖周报/月报迭代周期，过期自动失效；超出 7d 的财报可重新抽取（模型升级 / prompt 优化后值不同）
- **失败策略：Redis 故障静默 fallback 到 MQ**：`lookup`/`lookupAll`/`store` 全部 `onErrorResume` 静默吞掉 Redis 故障（返回 empty / empty Map / empty Mono）；上层 `checkCacheAndReplayOrDispatch` 走 `extractDispatcher.dispatchAll`，业务无感降级到原 MQ 路径
- **三表全命中才 replay，部分命中走 MQ**：避免半缓存导致数据不一致（BS 缓存 + IS 新抽 = 三表时间戳不一致，CHECK 勾稽会出问题）；保守策略宁可多抽一次也不冒数据不一致风险
- **`replayCachedExtracts` 不调 `extractTracker.recordSuccess`**：Redis 计数器专为 MQ 回报设计，cache replay 路径直接调 `transitionToExtractSuccess` 跳到 CHECK，绕过 `recordSuccess`/`hasFailed` 双查；也不调 `extractCacheService.store`，避免重复写缓存
- **`replayCachedExtracts` 标 step SUCCESS 复用 `transitionToExtractSuccess`**：与正常路径 `handleExtractStepSuccess` 的最终出口一致（`EXTRACT_SUCCESS` → `dispatchStepIfPending(CHECK)`），保证 task 状态机推进语义统一
- **`storeExtractCache` 在 `writeStatement` 之后**：避免 cache 写成功但 statement 写失败时下次 cache hit 拿到无 DB 行的"孤儿缓存"；spec §8.4 "失败不强制回滚" 意味着 statement 写失败不阻断，但 cache 写入应在 writeStatement 完成后才能保证一致性
- **`storeExtractCache` `onErrorResume` 兜底**：`reportRepo.findByTaskId` 抛异常时静默吞掉，不阻断主流程（extract 已成功，CHECK 触发路径仍可推进；缓存写入失败仅意味着下次重传仍走 MQ，业务无感）
- **`switchIfEmpty(Mono.defer(...))` 兜底 report 不存在**：历史 task 可能无 report 关联（如 M1.x mock 数据）；`flatMap` 在 empty source 上不触发会直接 `onComplete()`，`switchIfEmpty` 把这种情况兜底走 MQ
- **Mockito `lenient()` stub 默认 cache miss**：`TaskOrchestratorTest.setUp` 中 `extractCacheService.lookupAll` 默认返回空 Map、`store` 默认返回 empty Mono、`reportRepo.findByTaskId` 默认返回 empty Mono；个别 cache-hit 测试显式 `when(...).thenReturn(...)` 覆盖

### 已知限制
- **`lookupAll` 用 `concatMap` 顺序查 3 个 key**：3 次 Redis GET 串行（而非 `MGET` 批量）；3 表缓存查询 O(3) RTT，但 Redis 内网 < 1ms，整体仍比走 MQ 快 60s+；M3 性能优化时可换 `MGET` 批量
- **缓存按 step 独立 key，未做事务性批量写入**：`store` 三次独立 SET，理论上有 1-2 步成功 1 步失败的可能；但因 `lookupAll` 要求全命中才 replay，单步缺失会自然 fallback 到 MQ，不会读到部分缓存
- **`replayCachedExtracts` 标 step SUCCESS 不写 `started_at`/`duration_ms`**：缓存重放路径无实际 L3 推理时间戳，仅写 `finished_at`；这是 spec §3.10 的预期行为（cache hit 不该有 L3 推理时延统计）；运维如需区分 cache replay 与真实 SUCCESS，可查 `step.duration_ms IS NULL` 配合 `task.progress` 跳变识别
- **未接入 cache 失效信号**：当前 TTL=7d 自然过期；如未来需要主动失效（如 L3 模型升级 / prompt 优化），需新增 `extractCacheService.invalidate(pdfMd5)` 接口 + 触发点（M4 模型版本切换时）

### 设计决策
- **后端按 `statement_type` 分组查询（一次 3 次 flux）而非一次查全部**：`getStatements` 内 `findByReportIdAndStatementType` 调用 3 次（BS/IS/CF），让 Spring Data R2DBC 一次 SQL `WHERE report_id=? AND statement_type=?` 完成；比一次 `findByReportIdOrderByStatementTypeAscItemNameAsc` 再 Java 端分组更高效（R2DBC 返回 cursor 流式消费，分组需要全量物化）
- **归属校验在 Service 层而非 Controller**：spec §8.5 安全（用户隔离），AGENTS.md §3.1「禁止在 Controller 写业务逻辑」；`StatementQueryService.getReportDetail/getStatements` 用 `reportRepo.findById(reportId).filter(r -> r.getUserId().equals(userId))` 做归属校验，不归属返回 404 `REPORT_NOT_FOUND`（不暴露「报告存在但不归属」信息）
- **`ReportDetailResponse.createdAt` 转 Instant**：后端 entity `LocalDateTime` → DTO `Instant`（UTC）；前端 `formatTime` 用 `new Date(iso)` 兼容 ISO-8601；避免 LocalDateTime 序列化时区歧义
- **`StatementsResponse` 用 `List.copyOf` 防御性拷贝**：record compact constructor 把入参 List 包成不可变 List；SpotBugs `EI_EXPOSE_REP` 通过 `spotbugs-exclude.xml` 排除（record 已不可变，但静态分析无法证明）
- **前端 `displayValue(item)` 优先返回编辑值**：本地 `edited.values` map 优先于后端 `itemValue`；编辑后 input 边框变橙（`value-input--edited` class）+ 头部「已编辑 N 项」提示；plan M2.11 明确"暂不写回"，重置按钮清空编辑
- **`load(id)` 并发拉详情 + 三表**：`Promise.all([getReportDetail, getStatements])`；任意一个失败都进 catch，统一 `error.value` 处理；report 不存在时 `getReportDetail` 404，三表也 404，但前端只展示一次错误

- **千分位格式化但保留精度**：`formatNumber(value)` 用 `value.toFixed(4).replace(/\.?0+$/, '')` 去尾随 0，再 `toLocaleString('zh-CN', { maximumFractionDigits: 4 })`；编辑时 input 用 `displayValue` 展示千分位字符串，用户编辑后存原始字符串到 `edited.values`（不解析为 number，因为可能正在输入"1,234"中间状态）
- **`StatementTable` 用 `:class="{ 'value-input--edited': isEdited(row) }"`**：编辑后的 input 边框变橙（`box-shadow: inset 0 0 0 1px var(--fin-warning)`），背景淡橙；让用户一眼看出哪些已编辑
- **`ReportDetail.vue` 不订阅 SSE**：详情页是被动查看，不复用 TaskProgress 的 SSE；如解析未完成时进入详情页，三表为空 + Tab 内显示「暂无 X 数据，请等待 extract 阶段完成后再查看」占位

### 设计决策

- **Java IT 不起真实 RabbitMQ/L3 进程**：L3 真实推理需 GPU 跑 7B 4-bit 模型（~5GB VRAM，60s+），CI 不稳定；当前 `ai-service/app/modules/extractor/handler.py` 仍是 mock，不接真实 ModelHub。所以测试策略为：Java 侧用 `@MockBean TaskMessageProducer` 屏蔽 MQ；直接调 `TaskOrchestrator.handleStepProgress` 模拟 L3 回报，注入与 `extract_handler.handle` 返回结构一致的 fixture payload；真实 Redis（GenericContainer）验证 ExtractCacheService 全链路；真实 MySQL（MySQLContainer）验证 StatementWriter 三表写入与归属查询
- **Testcontainers Redis 用 GenericContainer 而非 testcontainers:redis**：maven central 找不到 `org.testcontainers:redis` 模块（不在 Testcontainers 官方支持列表）；用 `GenericContainer<DockerImageName.parse("redis:7.2-alpine")>` + `withExposedPorts(6379)` 替代，已有 `testcontainers:junit-jupiter` 依赖提供 `GenericContainer` 类
- **F1 评估脚本双模式**：`--mock-llm` 模式用 ground truth 自身作为模型输出，F1 必为 1.0，仅验证脚本可运行性；默认模式 HTTP 调 ai-service `/parse/upload` + `/internal/models/generate`，需 GPU + 已加载 7B 模型
- **三表 F1 计算口径**：item 名严格相等（case-sensitive）；value 相对误差 ≤ 1%（`math.isclose(predicted, truth, rel_tol=0.01)`）；TP/FP/FN 分别统计，precision = TP/(TP+FP)，recall = TP/(TP+FN)，F1 = 2PR/(P+R)；Overall F1 = macro-average of BS/IS/CF
- **SLA 脚本 EXTRACT 用单表 max 近似并行批次**：real 7B 模式顺序调 3 次 `/internal/models/generate` 测各表耗时，取 max 近似并行批次耗时（spec §12.1 视并行批次 ≤ 60s）；M2.08 真实 MQ 驱动的并行 EXTRACT 才是真实 SLA 来源，本脚本只是同步 HTTP 近似
- **SLA 脚本 CHECK/REPORT 标记为 not_exercised**：M2 阶段 L3 reasoner / generator 未通过 HTTP 暴露端点；spec §12.1 的 CHECK ≤ 30s / REPORT ≤ 45s 门槛在 IT 阶段无法触发；这些阶段留给 M3+ 接入 HTTP 后再覆盖
- **Java IT 用 `@ActiveProfiles("it")`**：IT 专用 profile 关闭 `spring.rabbitmq.listener.simple.auto-startup` 避免 RabbitMQ 自动连接失败导致 Spring 上下文启动失败；排除 `ReactiveSecurityAutoConfiguration` 跳过 SecurityFilterChain 装配（IT 不验证安全层）
